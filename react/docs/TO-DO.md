# 소셜 매칭 앱 — 관상 추천 엔진 & 피벗 설계

**최종 업데이트**: 2026-06-09

**피벗**: 관상 "리포트"를 만드는 앱 → 사람 풀에서 나와 맞는/닮은 TOP5를 추천하고 연결하는
**소셜 매칭 앱**. 관상은 *목적*이 아니라 *매칭 엔진의 연료*가 된다.

추천은 두 점수 레인으로 돈다:

1. **궁합 (compatibility)** — 나와 잘 맞는 사람 TOP5 (주 매칭 점수)
2. **닮은 꼴 (resemblance)** — 나와 비슷하게 생긴 사람 TOP5 (보조 발견 레인)

이 둘은 말은 비슷해 보여도 **수학적으로 완전히 다른 계산**이다. §0~§9 는 그 추천 엔진을
**고등학교 수학 지식만으로** 설명한다(`metrics` 에 vector 필드 추가). 피벗이 새로 요구하는
제품 축(풀·매칭·채팅·안전·프라이버시)은 바로 아래 "피벗" 절에 정리한다.

## 피벗 — 소셜 매칭 앱 (2026-06-09 결정)

### 확정한 제품 방향

| 결정 | 선택 |
|---|---|
| 포지셔닝 | **연애와 인맥 사이** — 순수 데이팅도, 순수 친구찾기도 아님 |

| 연결 수단 | **앱 내 채팅** — 연결 성사 시 인앱 메시지. 연락처 미노출 |

### 깨지는 핵심 가정 (= 출시 절대 게이트)

현 프라이버시 모델은 "UUID 추측 불가 = 링크 공유한 사람만 봄, 디렉토리 없음"(HOW-IT-WORKS
§12.4). 추천 풀은 **시스템이 능동적으로 내 얼굴을 모르는 사람에게 노출**한다 → 정반대.
얼굴은 PIPA 민감(생체)정보 → **별도 발견 동의·미성년 차단·차단/신고·삭제권**이 P0.
HOW-IT-WORKS §12 프라이버시 전면 개정 필요.

### 플로우 상세 (2026-06-09 확정)

**1) 등록·공유 = is_my_face 전용**
- metrics·R2 thumbnail 등록은 **is_my_face 인 경우만**. 앨범·남 얼굴 관상은 로컬 재미용(서버 0).
- 공유도 is_my_face 만 → 비-my-face 공유 시도 시 `WTF?! 내 얼굴만 공유할 수 있다.` 로 차단.
- is_my_face = **카메라 촬영만**(앨범 불가) — 가짜·도용·연예인 사진 1차 차단.

**2) 발견 풀 opt-in**
- 체크박스 카피: **"feeling kinda bored. 재밌는 거 뭐 없을까?"** (lonely 자기낙인 회피).
- 체크 → **동의 popup 한 번 더**(생체정보 능동 노출·철회 고지) → 풀 등록.

**3) mandatory intro form** (체크 시 미입력이면 강제 입력)
- 필드: **직업군 · 관심사 · MBTI**. (나이·성별은 분석 confirm 값 재사용)

**4) liveness 가드 (가짜사진 방지)**
- is_my_face 등록·재확인 시 **카메라로 얼굴을 다시 찍게 해 `shape_vec` 값을 비교** → 동일인 확인.
  기존 vector 인프라 재활용이라 구현 가벼움.

**5) 추천 엔진 — 두 레인** (= §0~§9 가 코어로 승격)
- `compat_vec`(궁합 MIPS) · `shape_vec`(닮은꼴 거리). 풀이 day-1 부터 커질 수 있어 옵션 1(필터)+
  옵션 3(embedding+재정렬)을 처음부터 함께 본다 (§5·§6).

**6) 알림 & 매거진 (retention·바이럴 엔진)**
- **닮은꼴 FCM**: 하루 1번. 단 **resemblance ≥ 80점** 후보가 있을 때만 발송, 없으면 침묵(억지 없음).
  → 닮은꼴을 0~100 점수화 필요.
- **궁합 천작지합(≥90.5) 커플 발견** → 1:1 특별 이벤트 알림.
- **weekly 웹 매거진**: blur 처리한 2인 페어 리스트 공개 + **"실제로 대화했는지" 여부까지 표기**
  → "주작 아님" 진정성·FOMO 장치. 정해진 발표 시간 없음(매거진 형태).

**7) 연결 = 양방향 동의 + 양쪽 과금** (상호 like 매칭 모델 대체)
- 추천에서 상대 프로필 보고 **대화 신청** → 선신청자 코인 지불, 상대에게 신청자 프로필 전달.
- 상대 **승낙/거절**. 승낙해도 **받는 쪽도 코인 지불**(일방 소비 아님). 선신청자가 더 냄, **비율 4:1~3:2**.
- 효과: 양쪽 skin-in-the-game → 스팸·어그로 강력 억제 + 받는 쪽 진심 담보. 새 테이블
  `chat_requests`(from·to·status·coin) · `messages`. 기존 `coins` 원장 재활용.

> **남은 미세 결정**: ① 거절·무응답 시 선신청 코인 환불 여부(소비=anti-spam vs 환불=공정)
> ② 코인 액수/비율 확정(예: 선 4 / 후 1) ③ intro form 에 "찾는 관계(연애/친구)" 포함 여부
> ④ TOP5 노출 방향(내 상위 vs 나를 상위에 넣은 사람)·알림 주기 트리거.

| 한 사람을 숫자 배열 하나로? | ✅ 그냥 됨 (z-score 배열) | ⚠️ 바로는 안 되지만 **변형하면 됨** (이 문서의 핵심) |
| DB가 바로 빠르게? | ✅ pgvector 인덱스 | 원래 느림 → 아래 방법으로 빠르게 |
| 저장 컬럼 | `shape_vec` (거리 비교) | `compat_vec` (짝 점수 비교) |

핵심 결론 두 줄:
- **닮은 꼴**은 쉽다. 얼굴을 숫자 배열로 만들고 "거리가 가까운 순서"로 정렬하면 끝. DB가 빠르게 해 준다.
- **궁합**은 원래 한 명 한 명 다 계산해야 해서 느린데, 궁합 점수의 **계산 구조가 특별한 모양**이라
  이걸 이용하면 궁합도 닮은 꼴처럼 "숫자 배열 + 빠른 검색"으로 바꿀 수 있다.

## 1. 준비운동 — 벡터·거리·내적 (고교 수학)

뒤 내용을 위해 딱 세 가지만 짚는다. 다 아는 내용이면 건너뛰어도 된다.

### 1.1 벡터 = 숫자를 순서대로 늘어놓은 것

좌표평면의 점 (3, 5)는 숫자 두 개를 순서대로 적은 것이다. 숫자를 더 많이 적으면 그냥 더 긴 줄이 된다:

```
얼굴 A = [1.1, -0.3, 0.8, 2.0, ...]   ← 얼굴 하나를 숫자 26개로 표현
```

이 "숫자 줄"을 **벡터**라고 부른다. 우리는 얼굴 하나를 숫자 26개짜리 벡터로 바꿀 것이다
(숫자 하나하나가 "코가 평균보다 얼마나 큰가" 같은 측정값 = z-score, §2).

### 1.2 거리 = 피타고라스 정리의 확장

두 점 (x₁, y₁), (x₂, y₂) 사이 거리는 피타고라스로:

```
거리 = √[ (x₁−x₂)² + (y₁−y₂)² ]
```

숫자가 26개여도 똑같다. 각 자리 차이를 제곱해서 다 더하고 루트:

```
거리(A, B) = √[ (a₁−b₁)² + (a₂−b₂)² + ... + (a₂₆−b₂₆)² ]
```

**거리가 작다 = 두 얼굴이 비슷하다.** 닮은 꼴은 이 한 줄이 전부다.

### 1.3 내적 = 두 벡터를 "자리끼리 곱해서 더한 것"

```
내적(A, B) = a₁·b₁ + a₂·b₂ + ... + a₂₆·b₂₆
```

내적은 "두 벡터가 같은 방향으로 얼마나 강하게 정렬돼 있나"를 재는 **하나의 숫자**다.
중요한 성질: **내적이 클수록 두 벡터가 잘 맞는다**고 해석할 수 있다.
궁합을 "내적이 큰 순서"로 바꾸는 게 이 문서의 목표다 (§4).

> 거리는 "차이를 제곱해서 더함"(작을수록 비슷), 내적은 "곱해서 더함"(클수록 정렬됨).
> 닮은꼴엔 거리, 궁합엔 내적을 쓴다 — 이 한 줄을 기억하면 된다.

## 2. 서비스 1 — 닮은 꼴 (쉬운 쪽)

### 2.1 얼굴 → 숫자 26개 벡터

엔진은 이미 얼굴마다 **z-score**를 계산한다. z-score는 "이 사람의 이 부위가 평균에서
표준편차 몇 칸만큼 떨어졌나"를 나타내는 숫자다.

- z = 0 → 딱 평균
- z = +2 → 평균보다 (그 그룹 기준으로) 꽤 큼
- z = −1 → 평균보다 살짝 작음

정면 측정 26개를 **항상 같은 순서**로 늘어놓으면 얼굴 하나가 26차원 벡터가 된다:

```
shape_vec = [faceAspectRatio_z, faceTaperRatio_z, ..., browSpacing_z]   (26개)
```

순서는 `face_reference_data.dart`의 `metricInfoList` 순서로 **영구 고정**한다. 순서가 바뀌면
저장된 모든 벡터가 의미를 잃는다.

### 2.2 TOP5 = "거리가 가까운 5명"

§1.2의 거리 공식으로 정렬하고 위에서 5명 자르면 끝이다. DB가 이걸 빠르게 해 준다 (pgvector).

```sql
-- 나(=$1 벡터)와 가까운 순서로 5명. 본인·이성 등은 미리 걸러냄
select id, alias, shape_vec <-> $1 as distance
from metrics
where gender_col = $2 and id <> $3
order by shape_vec <-> $1      -- <-> = 거리. 가까운 순 정렬을 인덱스가 가속
limit 5;
```

### 2.3 설계 결정 3가지

| 결정할 것 | 선택지 | 추천 | 이유 |
|---|---|---|---|
| 차원 수 | 정면 26개 / 정면+측면 34개 | **26개** | 측면(8개)은 일부 얼굴만 있어서 비면 거리가 망가짐. 모두가 가진 26개로 고정 |
| 원본 비율 vs z-score | 원본 비율 / z-score | **z-score** | z는 인종·성별 평균을 이미 빼서, 다른 그룹끼리 비교해도 공정 |
| 가중치 | 똑같이 / 코·눈 강조 | **똑같이** | "닮음"은 주관 개입 없이 자리마다 동등하게 보는 게 정직 |

## 3. 서비스 2 — 궁합 (어려운 쪽)

### 3.1 궁합 점수는 실제로 어떻게 나오나

코드(`compat_aggregator.dart`)상 궁합 총점은 **4개 부분 점수의 가중 평균**이다:

```
원점수 = 0.20 × 오행 점수    (얼굴형 5분류의 상생·상극)
       + 0.40 × 십이궁 점수  (12개 부위 "궁"의 짝 궁합)   ← 비중 가장 큼
       + 0.25 × 기질 점수    (오관 + 삼정 + 음양)
       + 0.15 × 친밀 점수    (남녀궁·입·눈)

총점 = 원점수를 사용자 눈높이 점수(0~100)로 살짝 펴 준 것
```

각 부분 점수는 "**두 사람을 서로 비교하는 규칙들**"의 합이다. 예를 들어 음양 점수
(`yinyang_matcher.dart`)는 이렇게 생겼다:

- 한 명은 양(陽)이 강하고 다른 한 명은 음(陰)이 강함 → **+18** (고전적으로 이상적인 짝)
- 둘 다 양이 강함 → **−12** (같은 자리 두고 부딪힘)
- 둘 다 음이 강함 → **−8**

### 3.2 왜 벡터 하나로 바로 안 되나

여기서 닮은 꼴과 결정적으로 다르다.

- **닮은 꼴**은 점수가 "A와 B의 거리"인데, 그 거리는 각자 벡터만 있으면 계산된다.
- **궁합**은 점수가 "**A가 양인가 음인가 × B가 양인가 음인가**"처럼 **둘의 조합**에 달렸다.
  게다가 "**서로 달라야 점수가 높은**" 보완 관계다 (양+음 = +18, 양+양 = −12).
  닮은꼴은 "비슷할수록 좋음", 궁합은 "잘 맞물릴수록 좋음" — 방향이 다르다.

그래서 닮은꼴 벡터(거리)로는 궁합을 절대 못 구한다.

### 3.3 O(N) 문제 — 왜 느린가

지금 방식으로 "나와 궁합 좋은 TOP5"를 구하려면:

```
풀에 있는 N명 전원과 1:1로 궁합 엔진(약 6,000줄 규칙)을 돌린다
→ N이 1,000명이면 1,000번, 100만 명이면 100만 번
→ 풀이 커질수록 선형으로 느려짐 (이게 "O(N)")
```

100만 명 풀에서 검색 한 번에 100만 번 엔진을 돌리는 건 현실적으로 불가능하다.

## 4. 궁합을 빠르게 — 핵심 통찰

> **한 줄 요약**: 궁합 점수의 계산 구조가 "**표(table)를 사이에 끼워 곱하는**" 특별한 모양이라,
> 이걸 쪼개면 **사람마다 숫자 배열(카드) 하나**로 바꿀 수 있고, 그러면 궁합 = **두 카드의 내적**이
> 되어 닮은꼴처럼 DB가 빠르게 찾는다.

### 4.1 궁합 점수 = "표 끼워넣기" (작은 예로)

오행 점수를 보자. 사람을 5종류(木·火·土·金·水)로 분류하고, 점수를 **5×5 표**에서 꺼낸다
(`element_matrix.dart` 실제 값):

| 내\상대 | 木 | 火 | 土 | 金 | 水 |
|---|---|---|---|---|---|
| **木** | 50 | 70 | 28 | 32 | 65 |
| **火** | 65 | 45 | 72 | 30 | 25 |
| **土** | 28 | 68 | 55 | 72 | 32 |
| **金** | 30 | 25 | 70 | 48 | 72 |
| **水** | 68 | 25 | 28 | 72 | 42 |

내가 木, 상대가 火면 → 표에서 **70**을 꺼낸다. 끝. 점수가 **오직 (내 종류, 상대 종류) 조합**에만
달렸다. 이런 걸 수학에서 **"표를 사이에 끼운 곱"** 이라고 부른다. 수식으로 쓰면:

```
점수 = (내 벡터) · 표 · (상대 벡터)
```

여기서 "내 벡터"는 내 종류만 1이고 나머지는 0인 배열이다. 내가 木이면 `[1,0,0,0,0]`,
상대가 火면 `[0,1,0,0,0]`. 이 둘을 표 양옆에 끼워 곱하면 정확히 표의 (木행, 火열) = 70만 살아남는다.

**중요한 발견**: 궁합 엔진의 네 부분이 *전부* 이 모양이다.

| 부분 점수 | 비중 | 두 사람을 어떻게 결합? | 표 모양 |
|---|---|---|---|
| 오행 | 0.20 | 5종류 × 5종류 | 5×5 표 |
| 십이궁 | 0.40 | 12개 부위 각각 "내 상태 × 상대 상태" | 부위마다 작은 표 |
| 기질 | 0.25 | 오관·삼정·음양 각각 "내 상태 × 상대 상태" | 작은 표들 |
| 친밀 | 0.15 | 남녀궁·입·눈 "내 상태 × 상대 상태" | 작은 표들 |

즉 **궁합 총점 = 큰 표 하나를 두 사람 벡터 사이에 끼운 곱**으로 통째로 쓸 수 있다.
(코드 근거: `palace_pair_matcher.dart`는 궁마다 점수가 `(내 상태, 상대 상태)` 쌍에만 의존하고
고정 가중치를 곱해 더한다 — 정확히 "표 끼워넣기" 구조.)

### 4.2 표를 쪼개면 "사람마다 카드 하나" = 내적

여기가 마법이다. 숫자 12를 3×4로 쪼개듯, **점수표도 두 조각으로 쪼갤 수 있다**
(수학적으로 대칭인 표는 항상 가능 — "고윳값 분해"가 보장한다. 이름은 몰라도 된다).

표를 쪼개고 나면, 각 사람에게 **"카드"라고 부를 숫자 배열 하나**를 줄 수 있고:

```
궁합(A, B) ≈ (A의 카드) · (B의 카드)        ← 그냥 내적! (§1.3)
```

비유하면:
- 원래는 "A와 B를 같이 놓고 두꺼운 규칙책을 펴서 점수를 매기는" 방식.
- 바뀐 뒤는 "A에게도 B에게도 **미리 계산한 숫자 카드**를 한 장씩 쥐여 주고,
  두 카드를 **곱해서 더하기만** 하면 궁합 점수가 (거의) 나오는" 방식.

이 "카드"가 바로 `compat_vec` 컬럼에 저장할 벡터다. 카드끼리 내적이 크면 궁합이 좋다.
그리고 **"내적이 큰 순서로 TOP5 찾기"는 DB(pgvector)가 빠르게 해 준다** — 닮은꼴과 똑같은 인프라,
거리(`<->`) 대신 내적(`<#>`)만 쓰면 된다.

### 4.3 카드를 어떻게 만드나 — 두 방법

**방법 A. 표를 직접 쪼개기 (해석적)**
엔진 구조에서 표를 그대로 조립해 수학적으로 분해. 정확하지만 부위·상태를 일일이 열거하는 수작업.

**방법 B. 흉내내기 학습 (distillation) — 추천**
우리는 이미 **Monte Carlo 시뮬레이터**(2만 쌍, seed=42)가 있다. 이걸로:

```
1. 가짜 얼굴 쌍을 수만 개 만든다
2. 각 쌍의 "진짜 궁합 점수"를 현재 엔진으로 계산해 정답표를 만든다
3. "카드끼리 내적 ≈ 진짜 궁합 점수"가 되도록 카드 만드는 규칙을 맞춘다 (회귀)
4. 잘 맞는지 점수로 확인한다
```

손이 덜 가고, 목표(검색 정확도)를 직접 최적화한다.

### 4.4 안전장치 — "대충 추리고, 정밀하게 다시 줄 세우기"

카드 내적은 진짜 엔진의 **근사치**다. 약간 틀릴 수 있다. 그래서 2단으로 한다:

```
1단계 (빠름, 근사): 카드 내적으로 후보 50명을 순식간에 추려온다 (pgvector)
2단계 (정밀, 정확): 그 50명에게만 진짜 궁합 엔진을 돌려 최종 TOP5를 뽑는다
```

**도서관 비유**: 사서가 제목만 훑어 관련 있어 보이는 책 50권을 빠르게 골라오고(1단계),
그 50권만 정독해서 최종 5권을 고른다(2단계).

이 구조의 핵심 장점: **카드는 완벽할 필요가 없다.** "진짜 TOP5가 추려온 50명 안에만 들어오면"
최종 순위·점수는 진짜 엔진이 보장한다. 그래서 근사 오차가 결과 정확도를 해치지 않는다.
우리가 감시할 지표는 단 하나 — **"진짜 TOP5가 추려온 50명 안에 들어오는 비율"(recall)**.

## 5. 옵션 비교 — 궁합 TOP-K 가속

N = 검색 대상 사용자 수.

| # | 방식 | 검색 속도 | 정확도 | 적합 규모 | 비고 |
|---|---|---|---|---|---|
| 0 | 전수 계산 (지금) | O(N), 매우 느려짐 | 정확 | ~수천 | 바닥. 풀 커지면 불가 |
| 1 | **싼 필터로 후보 컷** — 오행 5분류·성별·나이대로 먼저 거른 뒤 엔진 | O(걸러진 N) | 정확 | 어디든 깔림 | **거의 공짜, 학습 불필요.** 즉시 도입 |
| 2 | 사람마다 TOP-K 미리 계산해 저장 | O(1) 읽기 | 계산 시점엔 정확 | 풀이 고정적일 때만 | 새 얼굴마다 O(N) 쓰기 + **엔진 규칙 바뀌면 전부 무효화** → 우리 엔진 철학과 상극. **피함** |
| 3 | **카드(`compat_vec`) + 내적 검색 + 엔진 재정렬** | O(log N) + 50번 | **정확**(2단계 재정렬이 보장) | 수만~수백만 | 닮은꼴과 **같은 인프라**. 목표 아키텍처 |
| 4 | 사용자 행동 학습 랭커 | O(log N) | 데이터 의존 | 먼 미래 | 지금은 과함. 카드 컬럼이 그대로 진화 경로 |

## 6. 추천 로드맵

1. **지금 바로**: 옵션 1 — 오행 5분류·성별·나이대를 `body` JSON에서 **컬럼으로 빼서** WHERE로 후보를
   확 줄인 뒤 엔진. 거의 공짜이고 모든 단계에 깔린다.
2. **풀이 수만 넘으면**: 옵션 3 — Monte Carlo로 카드 만드는 규칙을 학습(§4.3 B) → `compat_vec`에 저장
   → 내적으로 후보 50명 → 그 50명만 진짜 엔진으로 재정렬(§4.4). 닮은꼴 인덱스와 같은 인프라라 추가 비용 작다.
3. 옵션 2(캐시)는 **피한다** — 우리 엔진은 규칙이 업데이트되면 과거 결과까지 새로 해석하는 게 정체성인데,
   캐시는 그때마다 전부 다시 계산해야 해서 상극이다.

## 7. `metrics` 스키마 변경 (벡터 두 컬럼)

`react/db/migrations/0001_baseline.sql` **직접 수정** (새 파일 추가하지 않음).

```sql
create extension if not exists vector;

alter table metrics
  add column shape_vec  vector(26),   -- 닮은꼴: z-score 26개, 거리(<->) 비교
  add column compat_vec vector(R),    -- 궁합: 학습된 "카드", 내적(<#>) 비교. R = 카드 길이(예 32~64)
  add column vec_version smallint;    -- 아래 ⚠️ 버전 태그

-- 닮은꼴: 거리 인덱스
create index on metrics using hnsw (shape_vec  vector_l2_ops);
-- 궁합: 내적 인덱스
create index on metrics using hnsw (compat_vec vector_ip_ops);
```

옵션 1(싼 필터)을 위해 `gender`·`ageGroup`·오행 5분류는 `body` 안이 아니라 **컬럼으로 승격**해야
WHERE가 인덱스를 탄다 (JSON 속 값은 인덱스가 안 걸린다).

> ⚠️ **버전 태그가 필요한 이유**: 두 벡터는 공유 시점에 한 번 계산해 **고정 저장**한다(인덱스라
> 매번 다시 못 만든다). 그런데 z-score는 reference 데이터에, 카드는 엔진 규칙에 의존한다.
> 나중에 reference나 엔진을 손보면 **저장된 벡터가 옛 기준**이 된다. 그래서 `vec_version`을 같이
> 적어 두고, 기준이 바뀌면 그 버전 행들만 골라 **다시 계산(backfill)** 한다. 이건 "저장은 raw만,
> 해석은 읽을 때 재계산"하는 Hive capture-only 원칙이 벡터에선 물리적으로 안 되는 지점이라 미리 못 박는다.

## 8. 검증 방법 (옵션 3을 켜기 전 통과할 것)

Monte Carlo 하니스로 측정한다.

- **recall@50** — 진짜 TOP5가 카드로 추려온 50명 안에 들어오는 비율. **이게 높으면 옵션 3은 안전**
  (최종 정확도는 2단계 엔진이 보장하므로). 목표 예: ≥ 0.99.
- **순위 상관(Spearman)** — 카드 내적 순위 vs 진짜 엔진 순위의 일치도. 참고 지표.
- **카드 길이 R 튜닝** — R을 키우면 근사가 정확해지지만 인덱스가 무거워진다. recall@50을 만족하는
  최소 R을 고른다 (예상 32~64).
