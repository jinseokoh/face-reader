## 1. 제품 개요

### 1.1 한 줄 정의

> **아는 사람들끼리 얼굴로 노는 관계 콘텐츠 앱.** 온디바이스 안면 계측(MediaPipe 468 landmarks)으로
> 관상을 수치화하고, 그 측정값으로 1인 관상 풀이 · 2인 궁합 · **다인 교감도(팀 N×N 케미 매트릭스)** 를 제공한다.

### 1.2 제품 분류 체계

| 제품 | 인원 | 역할 |
|---|---|---|
| **관상** | 1인 | 콘텐츠 층 — 26+8 metric → 14-node → 10 attribute → archetype → 8 인생 질문 본문 |
| **궁합** | 2인 | 五行·十二宮·五官·三停·陰陽 5-frame 별도 엔진, 4단 라벨 |
| **교감도** | 3~12인 | **1차 기능** — 지인 그룹(단톡방·회식·MT·가족)의 N×N 케미 매트릭스. 초대가 기능이 아니라 전제인 콘텐츠 |

용어 규칙: 문서·스토어 = `교감도`, UI 표면 = `그룹 · 케미 · 케미 그룹`, 코드 식별자 = 영문 `team_*`.
그룹 화면에서 멤버를 카메라로 등록하는 액션의 공식 표기는 **`직접촬영`** ("직접 등록"·"직접 스캔"·"카메라로 측정" 표기 폐기 — 2026-07-09 통일).

### 1.3 왜 교감도가 1차 기능인가 (기각 이력 요약)

| 검토안 | 기각 사유 |
|---|---|
| 시술상담·안경 피팅 유틸 | 해석 코퍼스 90% 사장, 더 나은 대안 실존 |
| 웹 결제 퍼널 | 무명 사이트 웹 결제는 신뢰 없음 → **결제는 전부 앱스토어 IAP** |
| 연예인 닮은꼴 | 부정경쟁방지법 + metrics 자체가 생체인식 특징정보(민감정보) |
| 일반인 닮은꼴 SNS / 공개 게시판 | 공유 준거 없음, 콜드 스타트, day-2 루프 부재. 모르는 타인 탐색은 본질이 아님 |

살아남은 구조: **"우리 팀 궁합표"는 혼자 존재할 수 없다** — 초대 자체가 바이럴 루프.
성별 무관 우정·케미 프레임이라 단톡방에 던질 수 있고(데이팅 거부감·4.3(b) dating 회피),
궁합 엔진의 pure 톤이 이미 비연애 페어용으로 구현되어 있다.

## 2. 목표와 성공 기준

### 2.1 비즈니스 목표

1. **App Store 4.3(b) 돌파** — 교감도라는 운세 템플릿 앱 어디에도 없는 1차 기능 + 온디바이스 468 landmark 측정 기술을 근거로 재제출. Android 선출시로 결제·바이럴 검증 후 iOS 재제출.
2. **설치 1명 → 영업사원 N명 구조** — 현장 경로(폰 1대로 전원 등록)와 원격 경로(카톡 초대 링크)가 각각 바이럴 루프를 만든다.
3. **코인 경제 단가 보호** — 매트릭스는 "대략 누가 잘 맞나"까지 무료, 숫자·풀이는 유료(1🪙). 1:1 궁합 단가와 충돌 없음.

### 2.2 제품 완성 기준 (출시 게이트)

- [ ] 2기기 시나리오 통과: A폰 생성 → 카톡 초대 → B폰 합류 → 양쪽 매트릭스 → 마감 → `/g/{id}` 웹 쇼케이스
- [ ] 현장 시나리오 통과: 생성 → 4명 등록 → 매트릭스(🏆 무료·내 행 고정) → 1🪙 unlock → 상세
- [ ] 웹 티저 실기기 통과: 카톡 인앱 → 외부 브라우저 탈출 → 카메라 → 즉석 결과 → 설치 유도
- [ ] 48h 자동 마감 서버 cron 동작 (방치 그룹이 영원히 열려 있지 않음)
- [ ] Android 프로덕션 출시 → iOS 재제출 패키지 제출

## 3. 타겟 사용자와 핵심 시나리오

### 3.1 페르소나

| 페르소나 | 상황 | 행동 |
|---|---|---|
| **방장(호스트)** | 회식·MT·동아리 모임에서 분위기를 만드는 사람 | 그룹 생성 → 그 자리에서 폰 하나로 전원 등록 → 매트릭스 공개 → 🏆 발표 |
| **원격 합류자** | 모임에 못 온 멤버, 단톡방에서 링크를 받은 사람 | 링크 탭 → 원탭 합류(또는 빈 슬롯 claim) → 자기 행이 매트릭스에 등장 |
| **미설치 수신자** | 카톡으로 링크만 받은 사람 | `/g/{id}` 웹 초대장 → **웹 티저 카메라로 즉석 맛보기** → 스토어 → 설치 → 카톡 재탭 |
| **솔로 사용자** | 자기 관상이 궁금한 사람 | 촬영/앨범 → 관상 리포트 → 공유 카드(`facely.kr/r/{id}`) |

### 3.2 대표 시나리오 — 회식 (현장 + 원격 혼합)

현장/원격은 **구현 경로일 뿐 사용자 선택지가 아니다.** 기본 시나리오가 혼합이다:
회식 자리에서 8명을 그 자리에서 직접촬영으로 등록하고, 못 온 2명은 카톡으로 초대한다.
그룹 화면에 [📷 직접촬영] · [💬 카톡으로 초대]가 **상시 공존**하는 이유.

### 3.3 초대 수신 요구사항 (3분기)

초대 링크를 받은 사람은 자신의 상태와 무관하게 **막히지 않아야** 한다:

| 수신자 상태 | 요구사항 | 구현 상태 |
|---|---|---|
| 앱 사용자 (내 관상 있음) | **내 관상을 원탭 전송**해 합류 — 재촬영 없음 | ✅ 구현 (TeamJoinScreen 원탭/슬롯 claim) |
| 앱 사용자 (내 관상 없음) | 셀카 등록 → 전송 | ✅ 구현 |
| 앱 미사용자 | React 웹(`/g/{id}`)에서 **맛보기 촬영** — 즉석 부분 결과 + 설치 유도 | ✅ 구현 (CameraTeaser) |
| 앱 미사용자 → 설치 전환 | **맛보기 촬영의 관상 데이터를 앱에서 재사용** — 설치 후 다시 찍지 않는다. 이를 위해 React 앱도 **카카오 로그인** 지원 | ☐ **미구현 (신규 요구사항, §4.1 웹 표면 참조)** |

## 4. 기능 요구사항

### 4.1 교감도 (1차 기능)

#### 그룹 생성
- 홈 [케미 그룹 만들기] → **바텀 슬라이드 풀페이지 스텝 플로우** (Toss 스타일):
  인트로 → 모임 훅 헤드라인("우리 〔팀/반/모임/…〕에서 나랑 케미가 제일 잘 맞는 사람은?" 키워드 회전)
  → 모임 유형 select(훅 문장 옵션 + 직접입력) → **나 포함 여부**("네, 나도 참여합니다" / "아니오") →
  참여 인원 슬라이더(3~12, 기본 6) → 즉시 그룹 화면.
- 빈 그룹도 가치가 있다 — 초대장이 곧 콘텐츠.

#### 인원 정책
**min 3 (2명은 1:1 궁합으로) · 권장 4~8 · 하드캡 12 (66쌍).**
캡 근거는 연산이 아니라 UX: 매트릭스 가독성(12열 = 가로 스크롤 한계), 현장 등록 시간(1인 30~60초 × 12 = 인내 상한), 하위 밴드 박제 리스크. 13명+ 대형 팀은 수요 관측 후 별도 설계.

#### 그룹 화면 (team_room)
- 헤더: 그룹명(기어 [그룹 설정]에서 수정) + "n/N명 등록" 진행바.
- 멤버 칩 그리드: 등록 완료 = 얼굴 아바타 / 대기 = 이름 + 점선 원(탭→등록). 방장 = gold 테두리 + "나" 배지.
- 액션: [📷 직접촬영](연속 루프: 촬영 → 성별·연령 확인 → 칩 추가 → 다음/끝) · 초대 3버튼(카톡/링크 공유/복사) 상시 공존.
- **케미는 발표(마감) 후에만 공개** (2026-07-11 확정 — 3명 부분 공개 로직 폐기: "N명이면 N명 온다"는 자연 기대와 카운트다운이 일치해야 한다). [그룹 케미 보기]는 마감된 그룹에서만. 방장은 3명 이상 등록 시 [마감하고 결과 발표]로 조기 발표 가능.
- **자동 완료**: 빈자리 없이 전원 등록(최소 3명) 시 `isFull` → `isClosed` **파생** — 저장 stamp 가 아니라 매번 계산이라 "정원 찼는데 열려 보이는" 상태가 구조적으로 불가능.
- 상태 3단: 모집 중 → 완료(마감) → 보관(30일 후, 미구현).

#### 매트릭스 화면 (team_matrix)
- 셀 = 4단 밴드 이모지: 💞 환상 케미 / 🔥 시너지 / 🤝 무난 / 🌧 보완 조합. **무료 셀에 점수·라벨 노출 금지.**
- **보는 사람 자신의 행을 항상 첫 행에 고정** (엑셀 틀 고정) — 사용자가 진짜 보는 건 66칸이 아니라 "나 vs 누구" 11칸.
- 표 위 요약 카드: 🏆 베스트 페어(점수 + 한 줄 풀이 **무료** — 도파민 모먼트) · 😲 의외의 조합.
- 나와의 교감도 순위 리스트(밴드 이모지만 무료, 탭 = unlock 플로우).
- 하위 밴드는 "보완 조합" 프레임 고정. 해시태그는 모던 톤(#케미 #호흡척척 #일치 계열).
- 계산: 궁합 엔진 N(N-1)/2회 호출 + 대칭 평균. 같은 멤버 구성 = 같은 결과(결정론).

#### 원격 합류 (카톡 초대)
- **lazy sync**: 그룹은 평소 로컬에만 산다. [카카오톡으로 초대] 첫 탭 시점에 서버 push. 평소 마찰 0.
- 서버 그룹 소유자 = **로그인 계정** (익명 초대 불가 — 소유가 흔들리면 안 됨). 초대 시 로그인 게이트.
- 초대 메시지: FeedTemplate(이미지 카드) + [참여하기] → `https://facely.kr/g/{groupId}`. 카톡 미설치 시 OS 공유 시트 fallback.
- 합류 방식 2가지: **(A) 빈 슬롯 claim** — 방장이 깔아둔 대기 이름("까불이")을 골라 그 행을 채움, 서버 키 `(team_id, name)`. **(B) 내 이름으로 새로 참여.**
- 반영은 폴링: 그룹 재입장(자동) 또는 pull-to-refresh(홈·그룹 화면).
- 점수 계산은 **앱에서만** — 서버는 그룹·멤버 명단과 (마감 시) `matrix_payload` 결과표만 보관. 궁합 엔진의 서버/웹 이식 없음.

#### 웹 표면 (`facely.kr/g/{id}` — 한 라우트, 두 얼굴)
- **마감 전 = 초대장**: 그룹명 · "N명 참여 중" · 참여자 칩 · 앱 열기/스토어 CTA + **웹 티저 카메라**:
  브라우저에서 정면 1장 → MediaPipe(client-only) → `runMetrics` → 방장 metrics 있으면 `runCompat`(나↔방장 점수), 없으면 `runEngine`(solo 한 입) → "정확한 결과는 앱에서". 카톡 인앱 웹뷰는 카메라가 막혀 있어 **외부 브라우저 자동 탈출** 후 재오픈.
- **마감 후 = 결과 쇼케이스**: 이름 + 밴드 이모지 매트릭스만. **사진·점수 없음** — 얼굴 썸네일은 방장 옵트인. 멤버가 동의한 건 그룹 안 공유지 전 세계 공개가 아니다.

#### 맛보기 촬영 데이터 재사용 (신규 요구사항, 미구현)

> 웹 티저에서 이미 얼굴을 찍은 사람에게 앱 설치 후 재촬영을 시키지 않는다 — 설치 퍼널의 마지막 마찰 제거.

- **React 앱 카카오 로그인**: Supabase Auth OAuth — 앱과 **같은 프로젝트·같은 계정 체계**. 티저 결과 화면에서 "이 결과를 앱에서 이어보기" 제안 → 카카오 로그인.
- **DeepFace 추정을 웹에도**: 앱과 동일 경로 재사용 — 촬영 프레임 1장을 R2 `temp/`(presign PUT, 기존 `/api/r2/presign`, 1일 자동 삭제)에 올리고 `analyze.facely.kr /analyze` 호출 → **성별·연령(·인종) default 추정**. 별도 입력 폼 없이 자동 prefill (사용자 수정 허용). 앱의 InfoConfirm 과 같은 패턴.
- **capture 귀속 저장**: 로그인 성공 시 `runMetrics` 결과(정면 26 metric) + DeepFace 추정 demographics 를 `metrics` 테이블에 `user_id` 귀속 + `is_my_face=true` 로 upsert.
- **앱에서 자동 복원**: 설치 후 같은 카카오 계정 로그인 → 기존 "로그인 rehydrate"(서버 metrics → 로컬 history 복원) 경로가 그 capture 를 내 관상으로 복원 → 그룹 원탭 합류. **재촬영 0회.**
- **측면 metrics 는 fallback**: 웹 캡처는 정면 1장 — 측면 8 metric 없이 저장하고 엔진은 그대로 수용(측면은 옵션 입력). 그룹 합류·궁합 계산은 정면 capture 만으로 성립. 측면 보완 촬영은 앱에서 선택(리포트 정밀도 향상 유도).

### 4.2 관상 (1인)

- 진입: 홈 [촬영]/[앨범] → 카메라(mesh overlay + 자동 카운트다운 2초, 정면 + 측면 yaw 45~80°) 또는 앨범(square-padding 처리) → 성별·연령·인종 확인(DeepFace 자동 prefill) → 리포트.
- 리포트: 10 attribute bar(탭 → top-5 contributor) · 음양 bar · 삼정 radar · 14-node expandable(밴드별 본문) · archetype + 8 인생 질문 본문 + special archetype.
- 본문 톤: **현대 한국어 평문** — 한자 단독 표기 금지, 메타포·자기계발 jargon 금지, 관찰→해석·행동 vignette·이중성 훅·2인칭 경험 예언·한 줄 평 6-레버.
- 공유: R2 thumbnail + Supabase upsert + `facely.kr/r/{uuid}` — 수신 측은 앱 딥링크 또는 웹 SSR 카드(같은 엔진을 JS 컴파일로 line-by-line 동일 렌더).
- 저장 원칙: **capture-only** — raw metric 만 저장, 점수·해석은 load 시 현재 엔진으로 항상 재계산. 엔진이 좋아지면 과거 리포트도 자동으로 좋아진다.

### 4.3 궁합 (2인)

- 두 리포트 짝짓기 → 총점 + 4 sub-score(五形和 0.20 / 宮位調 0.40 / 氣質合 0.25 / 性情諧 0.15) + 4단 라벨.
- intimacy 톤 3분기(pure/flirty/spicy — 성별·연령 조합으로 narrative 만 분기).
- 전체 본문 = 1🪙 unlock. 결제 시점에 두 사람 body 를 `unlocks` 에 동결(self-contained) — 서버 row 가 사라져도 구매한 궁합은 복원된다.

## 5. 비기능 요구사항

### 5.1 프라이버시·법적 가드레일

- **얼굴 사진은 온디바이스 처리 원칙.** 서버 업로드는 metrics(숫자) + 썸네일(선택)만. 원본 분석 이미지는 R2 `temp/` 1일 자동 삭제.
- metrics 는 생체인식 특징정보로 취급 — 연예인·실존 인물 비교 기능 영구 금지.
- 타인 등록 = 본인 캡처 동의 전제(등록 진입부 동의 UX).
- 웹 공개 뷰는 이름 + 밴드만 기본, 썸네일은 방장 옵트인.
- 링크 read = UUID 아는 사람(링크 공유 모델) — 유출 시 그룹명·멤버 이름 노출됨을 인지한 설계.

### 5.2 품질 게이트

- `flutter analyze` 0 issues · `flutter test` 전부 green (현재 151개).
- 엔진 룰·reference·quantile 변경은 `shared/` 한 곳에서만 + Monte Carlo 재보정(20,000 샘플, seed=42) 동반.
- UI 통일성 절대 1순위 — 디자인 토큰 SSOT(`flutter/docs/DESIGN.md`), 같은 역할 = 같은 토큰, 신규 색상 도입 금지, FontAwesome only.
- Hive·Supabase 스키마는 drop-recreate 자유(출시 전). DDL 은 `0001_baseline.sql` 단일 파일 직접 수정.

### 5.3 아키텍처 제약 (요약)

```
shared/ (face_engine, 순수 Dart SSOT) ── path dep ──▶ flutter/ (앱 셸)
        └── dart compile js -O1 ──▶ react/ (facely.kr Workers — share 카드 SSR + /g 초대장·쇼케이스·웹 티저)
python/ (DeepFace FastAPI — 성별·연령·인종 추정)     Supabase (metrics·coins·unlocks·teams·team_members)
```

- 궁합 엔진은 Flutter 에만 — react 는 `matrix_payload` 를 그리기만 한다.
- JS export 는 `runEngine` / `runCompat` / `runMetrics` (웹 티저용) 3개.
- 실시간 아님 — 폴링 기반 (의도된 결정: realtime 인프라 0).
