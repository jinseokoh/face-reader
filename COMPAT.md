# COMPAT.md — 궁합 엔진 로직 보고서

궁합이 어떻게 계산되고, 어떤 gate 를 거쳐, 어떤 톤으로 발표되는지의 전체 로직.
모든 용어는 실제 테이블·파일·상수 기준. 엔진 코드는 `shared/lib/domain/services/compat/`.

---

## 1. 저장 구조 — 뭘 저장하고, 뭘 저장하지 않는가

**궁합 점수·해설은 DB에 저장하지 않는다.** 저장하는 것은 "어느 얼굴 둘을 붙였는가"뿐이고,
결과는 화면을 열 때마다 그 자리에서 다시 계산한다. 엔진에 랜덤이 없어 같은 두 얼굴이면
언제 어느 기기에서 계산해도 점수·문장이 동일하고(결정론), A×B 와 B×A 의 총점도 같다(대칭).
그래서 엔진 로직을 고치면 마이그레이션 없이 모든 궁합이 새 기준으로 나온다.

| 계층      | 실제 이름                      | 실체                                                                            |
| --------- | ------------------------------ | ------------------------------------------------------------------------------- |
| 얼굴 원본 | `public.metrics`               | 얼굴 1회 캡처 = 1 row. row 의 UUID 가 그 사람 관상의 유일한 ID                  |
| 구매 기록 | `public.compatibilities`       | PK `(user_id, a_id, b_id)`. a_id·b_id 는 metrics UUID 둘, `check (a_id < b_id)` |
| 조회 키   | `uuid~uuid`                    | 두 metrics UUID 를 소문자·사전순 정렬 후 `~` 로 연결 (`compat_pair_key.dart`)   |
| 앱 모델   | `FaceReadingReport.supabaseId` | metrics UUID 를 Dart 쪽에서 부르는 이름                                         |

정렬해서 붙이므로 쌍에 방향이 없다 → 같은 두 사람은 1:1 이든 케미 배틀의 제3자 쌍이든
**한 번만 결제**(1코인, 구매자에게 영구). 재촬영은 my-face row 덮어쓰기라 UUID 가 안 변해
unlock 도 유지된다.

용어 규칙 (2026-07-21 rename, 1ed3a2a4): 엔티티(명사)는 `compatibilities` 로 통일
(테이블·`CompatibilityService`·provider), **unlock 은 "1코인 내고 여는 행위"를 가리키는
동사로만 남았다** (RPC `unlock_compat` · `CompatibilityService.unlock()` · coins
`'compat-unlock'`). 이 문서에서 unlock 은 전부 행위 의미다.

## 2. 계산 파이프라인 — 4층 (`compat_pipeline.dart`)

두 사람의 관상 리포트(zMap·nodeZ·faceShape·성별·연령대)를 넣으면 4개 부분 점수가 나온다.

| 입력  | 무엇인가                                                                                                                                                                                                                                                                             |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| zMap  | 개별 측정치(정면 17 + 측면 8)의 z-점수 map — `측정치 id → zAdjusted`. z 는 그 얼굴의 측정값이 동아시아 실측 평균(AAF reference)에서 표준편차 몇 개만큼 떨어져 있는지: 0 = 평균, + = 크거나 뚜렷, − = 작거나 약함. 50대+ 는 노화 편향을 빼고 쓴다(`age_adjustment.dart` 의 zAdjusted) |
| nodeZ | 얼굴을 14개 부위 트리(전체 → 구역 → 말단 부위)로 묶고, 부위마다 소속 측정치들의 z 평균(`ownMeanZ`)을 낸 값 — `부위 id → 평균 z`. 개별 측정치보다 한 단계 굵은 "부위 단위 강약"으로, 십이궁(L2) 판정의 재료                                                                           |

예: `lipFullnessRatio` 의 z 가 +0.8 이면 "입술이 평균보다 꽤 두툼"이고 이건 zMap 에서
L4(입술 geometry)가 직접 읽는다. 반면 "눈 부위 전체가 강한가"는 nodeZ 의 눈 노드 평균으로
읽는다 — 같은 얼굴 데이터를 해상도만 달리 본 것이다.

| 층  | 내부 이름      | 보는 것                                                | UI 표기     | 총점 가중치 |
| --- | -------------- | ------------------------------------------------------ | ----------- | ----------- |
| L1  | 오행(五形)     | 얼굴형 → 나무·불·흙·쇠·물 분류 후 상생·상극 관계       | 가치관      | 0.20        |
| L2  | 십이궁         | 결혼·가족·재물 등 12개 궁 부위의 강약 짝               | 관심사      | **0.40**    |
| L3  | 기질           | 눈·코·입 짝(0.55) + 얼굴 3구역(0.25) + 음양 균형(0.20) | 소통 스타일 | 0.25        |
| L4  | 성정(intimacy) | 남녀궁·처첩궁 짝 + 입술 모양 + 눈매                    | 이성적 끌림 | 0.15        |

각 부분 점수는 50 을 기준으로 가감 후 5~99 로 자른다. L1 은 방향에 따라 점수가 달라
양방향 평균으로 대칭화한다.

L4 세부 — 4개 축과 델타 범위: 남녀궁 ±18 / 처첩궁 ±18 / 입술 ±14 / 눈매 ±10.
점수식은 `50 + 네 축 합`, clamp 5~99. 모든 페어에서 계산·노출된다.

## 3. 총점 — 분포를 점수로 번역 (`compat_aggregator.dart`)

```
rawTotal = 0.20·L1 + 0.40·L2 + 0.25·L3 + 0.15·L4
spread   = clamp(50 + (rawTotal − 50) × 1.4, 5, 99)     ← 가중평균 수축 보정
total    = remap(spread)                                 ← 아래 anchor 로 구간별 선형 변환
```

remap anchor (20k 쌍 몬테카를로, seed 42 의 p30/p60/p90 실측):

| spread        | 5   | 50.54 | 54.44 | 59.63 | 99  |
| ------------- | --- | ----- | ----- | ----- | --- |
| **화면 점수** | 0   | 56    | 78    | 90    | 100 |

즉 화면 점수는 절대값이 아니라 **전체 분포에서의 위치**다 — 상위 10% 쌍이 90점 이상이
되도록 설계. 4개 부분 점수도 같은 문법의 display-only remap 을 거친다
(`compat_sub_display.dart`, 내부 가중합은 raw 그대로).

## 4. 등급 gate (`compat_label.dart`)

발표문의 뼈대 톤을 결정하는 최상위 분기. 목표 분포 10/30/30/30.

| 등급                | gate (total) | headline                                          |
| ------------------- | ------------ | ------------------------------------------------- |
| 천생연분 (天生緣分) | ≥ 90.5       | 얼굴로 읽으면 흔치 않게 궁합이 잘 맞는 관계       |
| 금슬화합 (琴瑟和合) | ≥ 81.5       | 서로 결이 잘 맞아 자연스럽게 화목한 관계          |
| 상부상조 (相扶相助) | ≥ 61.5       | 노력해서 맞춰 가야 완성되는 관계                  |
| 형극난조 (刑剋難調) | 미만         | 자주 부딪힐 소지가 크니 조심해서 지켜야 하는 관계 |

등급은 갈등 시나리오·전략 섹션의 도입/마무리 문장 pool 과 등급별 기본 전략 1개
(유지 / 표현·조율 / 과정 집중 / 규칙화)를 결정한다.

## 5. 발표문 구성 (`compat_narrative.dart`)

고정 5섹션 + 별도 챕터 1. 문장 variant 는 두 UUID 의 해시(pair-seed)로 골라 **같은 쌍은
항상 같은 문장**.

| 섹션               | 무엇을 고르나                                   | 구조                                                         |
| ------------------ | ----------------------------------------------- | ------------------------------------------------------------ |
| 1. 한줄 요약       | 오행 관계 5종별 고정 펀치라인                   | "좋은데 이게 위험" 이중성 + headline + 점수·등급 + 오행 구도 |
| 2. 핵심 궁합 3가지 | 전체 발견을 중요도(\|delta\|×priority) 순 top 3 | 의미 / 실제 모습 / 장점 / 주의할 점                          |
| 3. 갈등 시나리오   | 부정 발견 중 delta 낮은 순 top 3                | 등급별 도입 → (근거·실제 모습·확대 양상)×3 → 등급별 마무리   |
| 4. 운영 전략       | 등급 기본 1 + 부정 top 2 + 긍정 top 1           | action / 근거 / 실행 방법 / 실패하는 경우                    |
| 5. 점수+이유       | 부분 점수 4개 정렬                              | 최강 축("지탱") / 최약 축("갈등이 먼저 터짐") / 등급 tier    |
| +끌림 챕터         | §6 gate                                         | 5섹션 계약 밖, UI 에서 별도 렌더                             |

한줄 요약의 오행 관계 5종: 상생(내가 낳음)·피생(상대가 받쳐줌)·상극(내가 리드)·
피극(상대가 다잡음)·비화(닮은꼴). 오행이 애매하면(confidence < 0.08) "나무·흙 겸형" 표기.

## 6. 이성적 끌림 챕터 — 톤·버킷 gate (`intimacy.dart` + narrative)

점수식은 모든 페어 동일, **어휘 수위만** 분기. 위에서부터 먼저 맞는 조건 적용:

| 순서 | 조건                          | 톤                              |
| ---- | ----------------------------- | ------------------------------- |
| 1    | 동성 페어                     | pure — 점잖은 산문              |
| 2    | 한쪽이라도 10대 또는 70대+    | pure                            |
| 3    | 이성 페어 + 양쪽 모두 30~50대 | spicy — 홈런을 염두에 둔 분위기 |
| 4    | 그 외 (이성 + 한쪽 20대/60대) | flirty — 미묘한 긴장            |

pure/flirty 는 opener 1문단 + closer 1줄만 톤별이고 축 본문 4문단은 공통 pool 을 쓴다.
**spicy 는 챕터 전체가 전용 pool 이다** (2026-07-22 v2) — opener 24 + closer 24 에 더해
spicy 전용 축 산문 24개(성별 2 × 축 4 × 부호 3)를 따로 가져, opener 부터 closing 까지
한 편의 기사로 읽힌다. 전부 `compat_phrase_pool.dart` 의 const map.

| gate     | 기준                                  | 효과                                       |
| -------- | ------------------------------------- | ------------------------------------------ |
| 버킷     | raw L4 ≥65 high / 45~65 mid / <45 low | 여는·닫는 문장 pool + 조언 강도            |
| 축 부호  | 축 값 ≥+3 pos / ≤−3 neg / 사이 neu    | 축별 문단 선택 (pos/neu/neg 모두 pool 존재) |
| mid 조언 | pair-seed 짝수 → low 쪽 조언          | 낮은 점수에서 들이대는 문장 방지           |
| 시점     | 보는 사람 성별 기준 pool              | 조언 어휘 분기                             |

### spicy v3 — 욕망 폭로 내러티브

직장인 상황극이 아니다. 직업·소속·장소를 전제하지 않고, 누구라도 대입할 수 있는
**보편적인 욕망과 자기검열**을 다룬다. 내러티브 엔진은 두 개:

1. **금지된 상상 폭로** — 독자가 상대를 보며 해 봤지만 인정 안 하던 상상(둘만 남는
   밤, 그다음 장면)을 직접 짚는다.
2. **본능 폭로** — "설렘" 같은 추상어 대신 실제 반응을 쓴다: 거리 의식, 시선이
   머무는 지점, 목소리·향 의식, 대화를 끝내기 싫음, 옷차림 변화를 먼저 알아챔.

핵심 감정 문장: **"이 사람은 한번 살을 맞대고 싶은 사람이다"** — 무작위 삽입이 아니라
`자기 반응 → 숨긴 상상 폭로 → 호감으로 설명 안 되는 끌림 → 욕망의 직접 인정 → 행동
촉구` 흐름의 결론으로만 등장한다.

- **opener 5단 공식** — `{X}` 로 욕망 상태 단정 → 상상 폭로 → 본능 반응 → 욕망 인정
  → 버킷별 행동 방향.
- **closing** — 행동 촉구 장치 (후회 회피 / 약속 제안 / 긴장 인정 / 성인다운 직접성).
- **금지 3종** — ①직업 특정 장면(회식·퇴근·엘리베이터·사무실·출퇴근), ②사소한 행동
  과잉 해석("버튼 늦게 눌렀다=마음 있다" 류 억지 추리), ③추상 감정 문장(묘한·왠지
  모르게·운명처럼·설레는 마음).
- **수위** — 홈런·대시·속궁합·살을 맞대다·침대·호텔·밤 허용. 실제 성행위·신체 행위의
  구체 묘사 금지 — 핵심은 행위가 아니라 행동 직전의 긴장감과 먼저 움직일 용기.
- **POV 분리** — 남성 map 은 본능 반응 폭로 → 직선 행동 권유, 여성 map 은 신호 해석 →
  본인의 주도권 회수. 문장 골격 자체가 다르다.

| 버킷 | 긴장감 계약                                                      | 재료 예                                        |
| ---- | ---------------------------------------------------------------- | ---------------------------------------------- |
| high | 이미 충분히 가까움 — "왜 아직 아무것도 안 했지?". closing 최과감 | 둘만 남는 밤의 상상, 되감기하는 장면, 스친 팔  |
| mid  | 욕망은 있고 결정타 없음 — 반응을 확인할 행동 권유                | 어깨 거리 계산, 화제 하나 더 꺼내기, 거울 습관 |
| low  | 일방적 관심 — 진행된 관계로 쓰지 않는다. "홈런 예고 대신 출루"   | 끌림인지 호기심인지 판정, 커피 한 잔부터       |

### 점수 ↔ 발표문 연계 요약

| 점수                    | 발표문의 어디를 움직이나                                             |
| ----------------------- | -------------------------------------------------------------------- |
| total                   | 등급 → headline, 갈등·전략 섹션의 도입/마무리 pool, 등급별 기본 전략 |
| 부분 점수 4개 (display) | §5 섹션5의 최강 축("지탱")·최약 축("먼저 터짐") 지목 + 해시태그 chip |
| raw L4 (끌림)           | 버킷 → 끌림 챕터 opener/closer 수위, `{X}` 로 문장 안에 직접 삽입    |
| L4 축별 delta           | 부호 → 어떤 축 문단이 실리는지                                       |
| 발견별 delta            | 핵심 3가지·갈등 시나리오 3개·전략 2+1 의 선별 순서                   |

## 8. 케미 배틀 발표 gate (`battle.dart` + `team_reveal_screen.dart`)

모든 쌍 N(N−1)/2 회 엔진 호출 후 정렬. 정렬 순서가 곧 케미 순위.

| gate      | 기준                                                         | 효과                                                                  |
| --------- | ------------------------------------------------------------ | --------------------------------------------------------------------- |
| 무료 노출 | payload 는 쌍마다 등급(band)만, 점수는 best 한 쌍만          | "숫자·풀이 = 유료" 정책                                               |
| 차단 쌍   | total 을 60.0 으로 눌러 형극난조 확정                        | 베스트에 못 오르고, 상대에겐 "궁합 나쁨"으로만 보여 차단 사실이 안 샘 |
| 베스트    | total 내림차순 → slot 오름차순, blocked 제외, 공동 수상 없음 | 연출이 항상 한 쌍만 가리킴                                            |
| 매칭 모드 | 이성 쌍만 계산 (동성 쌍은 목록에 없음)                       |                                                                       |
| 쌍 상세   | 기존 unlock 흐름 재사용 — 1코인, 기구매 쌍은 무료 통과       | unlock 직전 분석 실행해 total 을 admin 용 기록                        |

## 9. 문체 원칙 (narrative 공통)

- 본문에 한자 금지, 현대 한국어만 (등급명 괄호 한자는 예외).

## 10. 알려진 불일치 · 개선점

1. **등급 gate 불일치** — 등급은 **90.5 / 81.5 / 61.5** 인데 해시태그 비율 gate 와
   aggregator 주석은 옛 경계 **90 / 78 / 56** 을 쓴다. 경계 구간의 쌍은 등급과 태그
   비율이 한 단계 어긋날 수 있다.
2. **clamp 조기 발동** — ×1.4 확대 때문에 rawTotal ≤17.86 은 전부 0점, ≥85 는 전부
   100점으로 뭉개진다. 실측 분포상 도달 사례는 사실상 없으나 설계상 알고 있을 것.
