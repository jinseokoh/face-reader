# 케미 배틀 — 방 만들기 재설계 + 베스트 매칭 성사 UX

**작성일**: 2026-07-17
**성격**: UX 설계 문서 (구현 계획 선행). 2026-07-16 설계 문서 위에 제품 오너 최종 요구사항을 반영해 생성 플로우와 결과 이후 플로우를 다시 그린다.
**토큰 준수**: `flutter/docs/DESIGN.md` 전면 준수 — 신규 색 금지, chip 단일톤, CTA 흰 배경 + 1px textPrimary border, SongMyung 은 `AppText.display` / `appBarTitle` 한정, 아이콘은 FontAwesome + 보유 asset(`assets/icons/male.png`, `female.png`)만.

## 0. 이번 재설계가 폐기·대체하는 것

| 폐기 | 대체 |
|---|---|
| 공약(pledge) 스텝 전체 + `pledge` 컬럼 + 공약 회수 카드 | 제목 프리셋의 "천작지합 궁합이면 ~한다" 카테고리가 재미를 흡수 (§C) |
| `chat_url` (카카오 오픈채팅 수동 링크) | 베스트 매칭 상호 동의 → 인앱 1:1 채팅 (Supabase, §E) |
| 방 제목 자유 입력 (24자 TextField) | 카테고리 → 제목 프리셋 2단 선택 (§C). 자유 텍스트 모더레이션 표면 소멸 |
| 인원 4~12 스테퍼 | 6 / 8 / 10 / 12 네 개 고정 선택 (이성방 반반과 정합) |
| 연령 RangeSlider(10~70, 전연령 포함) | 자기 나이대 포함 인접 범위 버튼 2개만 활성 (§B). 전연령 옵션 소멸 |
| 단일 방 유형 | 방 유형 2종: 이성 케미 매칭방 / 전체 케미 배틀방 (§A.2) |

**명시적 제품 결정 — 연령 하한 20세**: 생성 가능 연령 범위의 하한을 20세로 고정한다
(§B 옵션 리스트가 20부터 시작). 모든 방의 `age_min >= 20` 이 성립하므로 **10대는 생성뿐
아니라 참가도 구조적으로 불가**하다. 근거: 이 기능은 사진 상호 공개 + 1:1 채팅으로
이어지는 매칭 표면이고, 미성년자와 성인이 한 방에 섞이는 경로를 데이터 규칙 수준에서
차단하는 것이 UI 게이트보다 강하다. (요구사항은 "생성 불가"까지만 명시했으나, 하한 20
결정의 자연 귀결로 참가 불가가 따라온다 — 의도된 결과로 확정한다.)

## A. 생성 플로우 재설계

### A.0 진입 게이트 — 10대 차단은 스텝이 아니라 문 앞에서

**위치**: 케미 탭의 [배틀 만들기] 탭 시점. bottom sheet 를 열기 전에 내 관상의
`ageGroup` decade 를 확인하고, 10대면 시트를 열지 않고 안내 다이얼로그를 띄운다.

- 근거: 스텝 중간(연령 스텝)에서 막으면 사용자가 제목까지 고른 뒤 버려지는 최악의
  낭비가 생긴다. 진입 시 차단이 유일하게 정직한 위치다.
- 다이얼로그 (§3.1 모달 표준 — white bg, `AppRadius.xl`, title `modalTitle`, body `body`):

> **제목**: 방 만들기 사용불가
> **본문**: 케미 배틀 방 만들기는 20세 이상부터 사용할 수 있습니다. 내 관상 분석의
> 나이대가 10대로 확인되어 지금은 만들 수 없습니다.
> **액션**: [확인] (TextButton, textPrimary)

- 참가 딥링크로 들어온 10대는 RPC `AGE_NOT_ALLOWED` 가 막는다 (현행 에러 계약 재사용,
  카피: "이 방의 연령대에 해당하지 않습니다").

### A.1 스텝 구조 — 5스텝, 한 화면 한 질문

```
① 방 유형  →  ② 방 제목  →  ③ 인원  →  ④ 연령대  →  ⑤ 공개 설정  →  [배틀 만들기]
```

- 현행과 같은 5스텝을 유지하되 질문이 전부 교체된다. 스텝 수를 4로 줄이는 안
  (제목+유형 합침)은 기각 — 유형 선택이 제목 프리셋 목록을 필터하므로(§C) 유형이
  반드시 선행 확정되어야 하고, 한 화면에 두 선택을 얹으면 필터가 눈앞에서 바뀌는
  혼란이 생긴다.
- ⑤만 예외적으로 두 질문(공개 방식 + 썸네일 공개)을 담는다. 둘 다 "이 방에서 무엇이
  공개되는가"라는 단일 주제이고, 각각 독립 스텝으로 쪼개면 7스텝이 되어 이득이 없다.
- 상단 진행 표기: 현행 `n / 5` (`AppText.hint`) 유지. 뒤로가기 arrowLeft FaIcon 유지.
- 하단 CTA: `PrimaryButton` — ①~④ 는 "다음", ⑤ 는 "배틀 만들기". 각 스텝의 유효성
  미충족 시 disabled (흰 bg + border 톤 테두리 + textHint — PrimaryButton 잠긴 토큰).

### A.2 스텝 ① 방 유형

- 타이틀: `AppText.display` "어떤 방을 만들까요?"
- 컴포넌트: 현행 `_choiceTile` 2개 (surface bg, 선택 = 1px textPrimary border,
  비선택 = border 톤). 기본 선택 없음 — 유형이 이후 모든 스텝을 좌우하므로 명시 선택을
  요구한다. 미선택 시 [다음] disabled.

| 타일 | title (`subTitle`) | caption (`caption`) |
|---|---|---|
| 이성 케미 매칭방 | 이성 케미 매칭방 | 남녀 자리가 반반으로 고정됩니다. 결과는 남녀 쌍만 계산합니다 |
| 전체 케미 배틀방 | 전체 케미 배틀방 | 성별 구분 없이 모든 쌍의 케미를 계산합니다 |

- 이성 케미 매칭방 타일 하단 보조 문구 (선택 시에만 노출, `caption` + textHint):
  "베스트 매칭이 되면 두 사람에게 서로의 사진이 공개되고 채팅을 제안할 수 있습니다"
  — 방장이 방의 성격을 만들기 전에 알게 한다.

### A.3 스텝 ② 방 제목 (2단 선택 — 상세 §C)

- 타이틀: `AppText.display` "방 제목을 고르세요"
- 보조: `caption` "방 목록과 초대장에 그대로 보입니다" (현행 카피 재사용 — 같은 정보
  같은 포맷).
- 구성: 카테고리 chip 가로 스크롤 행 → 아래 제목 리스트 (§C.3). 자유 입력 없음.
- 유효성: 제목 1개 선택 시 [다음] 활성. 카테고리만 고른 상태는 미완.
- 방 유형의 영향: 현재 유형에 허용되지 않는 제목은 목록에서 **제외** (disabled 나열이
  아니라 미노출 — §C.2 근거).

### A.4 스텝 ③ 인원

- 타이틀: `AppText.display` "몇 명이 참가하나요?"
- 보조: `caption` "정원이 다 차면 배틀이 자동으로 시작됩니다" (현행 유지).
- 컴포넌트: 단일톤 chip 4개 가로 배치 — [6명] [8명] [10명] [12명]. 기본 선택 8명
  (현행 기본값 승계). 스테퍼(±) 폐기 — 유효값이 4개뿐인데 연속 조작 컨트롤은 과함.
- **이성방일 때**: 선택된 chip 아래 `body` 로 반반 구성을 즉시 표기.
  예: 8명 선택 → "남자 4명, 여자 4명". 전체방이면 이 줄 없음.
- 상태: 항상 유효 (기본 선택 존재) → [다음] 상시 활성.

### A.5 스텝 ④ 연령대 (상세 §B)

- 타이틀: `AppText.display` "참가 연령대"
- 보조: `caption` "방장의 나이대가 포함된 범위만 고를 수 있습니다"
- 컴포넌트: 범위 버튼 5개 세로 리스트, 활성 최대 2개 (§B). 기본 선택 없음 —
  선택 시 [다음] 활성.

- 타이틀: `AppText.display` "공개 설정"

## B. 연령 선택 UI — 슬라이더 폐기, 범위 버튼

### B.1 결정: 고정 5개 범위 버튼, 활성 최대 2개

RangeSlider 를 폐기하고 아래 5개 버튼의 세로 리스트로 교체한다.

```
[ 20~39세 ]   ← 방장 20대: 유일 활성 / 30대: 활성
[ 30~49세 ]   ← 방장 30대: 활성 / 40대: 활성
[ 40~59세 ]
[ 50~69세 ]
[ 60~79세 ]
```

각 버튼 = 인접 두 decade 쌍 (`age_min`, `age_max = age_min + 10`). 방장 decade 가
포함된 버튼만 활성 — 중간 나이대는 2개, 20대와 70대는 1개 활성.

**근거 (슬라이더 기각 이유)**:
1. 유효 선택지가 최대 2개다. 연속 범위 컨트롤은 "고를 수 있는 것처럼 보이는" 6개
   division 중 4~5개가 죽어 있는 기만적 UI 가 된다.
2. 슬라이더는 disabled 구간을 표현할 표준 문법이 없다 — 드래그를 막으면 고장으로
   읽히고, 스냅백은 에러 안내를 반복시킨다. 버튼의 disabled 는 한눈에 읽힌다.
3. 요구사항이 "선택 불가 옵션은 disabled 로 보이게"다 — 전 옵션을 나열하고 흐리게
   하는 것은 리스트형 컨트롤의 본성이다. 전체 스펙트럼이 보이므로 "왜 이것만 되는가"가
   보조 문구 한 줄로 설명된다.

### B.2 상태 표현

컴포넌트는 `_choiceTile` 과 동일 레시피 (surface bg, `AppRadius.lg`), 라벨은
`_ageRangeLabel` 포맷 그대로 ("20~39세" — 같은 정보 같은 포맷 원칙).

| 상태 | 표현 |
|---|---|
| 활성 미선택 | surface bg, 1px border(#E0E0E0), 라벨 textSecondary |
| 선택 | 1px textPrimary border, 라벨 textPrimary w600 |
| disabled | surface bg, border 톤 유지, 라벨 textHint. 탭 무반응 (스낵바 등 소음 없음) |

- disabled 버튼 오른쪽에 부가 설명을 넣지 않는다 — 리스트 하단 한 줄이면 충분:
  `caption` + textHint "방장 나이대(30대)가 포함된 범위만 선택할 수 있습니다"
  (괄호 안은 방장 decade 실값).
- 기본 선택 없음. 활성이 1개뿐이어도 자동 선택하지 않는다 — "이 방은 이 연령대"라는
  선언은 방장이 한 번 누르게 한다 (한 탭 비용, 명시성 이득).
- 전연령 옵션은 없다 (§0 하한 20 결정과 함께 소멸).

## C. 제목 프리셋 카탈로그

### C.1 원칙

- 자유 입력 전면 폐지. 제목 = 카테고리 1개 + 프리셋 1개 선택.
- 카피 규칙: 감정·효과 단정 금지("재밌어요" 류 없음), 저급 슬랭 금지, 가운데점 금지,
  현대 한국어. "천작지합 궁합이면 ~한다" 형은 구 공약의 재미를 흡수하는 약속형 제목 —
  실행 강제 장치는 없고 방의 정체성 선언으로만 기능한다 (문서상 명시; UI 에 별도
  면책 문구는 넣지 않는다).
- 허용 유형: 각 제목은 `공용` / `이성방` / `전체방` 중 하나. UI 는 현재 방 유형에
  허용되지 않는 제목을 **숨긴다** (disabled 나열 기각 — 고를 수 없는 문장 목록은
  스크롤 소음일 뿐이고, "전용" 라벨을 chip 안에 넣으면 단일톤 규칙과 충돌한다).

## D. 로비와 결과표 — 유형별 차이

### D.1 이성방 로비 — 남녀 2열 슬롯

```
서울지역 케미 배틀            ← appBarTitle
20~39세  ·없음·  [남녀 반반 8명]   ← 메타 줄 (아래 참조)

  남  2 / 4         여  1 / 4      ← 열 헤더
 ┌─────────┐      ┌─────────┐
 │ ◉ 민준   │      │ ◉ 지은   │    ← 채워진 슬롯
 │ ◉ 철수   │      │ ◌       │    ← 빈 슬롯
 │ ◌       │      │ ◌       │
 │ ◌       │      │ ◌       │
 └─────────┘      └─────────┘
        [QR]  [초대 버튼들]
```

- **두 열, 색 구분 없음** — 신규 색 금지 + 단일톤 원칙. 성별 구분은 (1) 열 위치,
  (2) 열 헤더 (male.png / female.png 16px + "남 2 / 4" `sectionTitle`), (3) 빈 슬롯
  안의 성별 아이콘, 세 겹으로 충분하다.
- 메타 줄 포맷: 연령대와 인원 구성은 줄을 분리해 표기 (가운데점 금지):
  1줄 "20~39세", 2줄 "남녀 반반 8명". 전체방은 2줄 "8명".
- **채워진 슬롯**: 42px 원형 아바타 + 닉네임 `caption`. 아바타 소스는 §D.4 규칙.
  입장 시 flip-in 연출 유지 (현행).
- **빈 슬롯**: 42px 원, 1px border(#E0E0E0), 내부에 해당 열의 성별 아이콘
  (male.png / female.png) alpha 0.35 — "이 자리는 남자 자리"가 아이콘으로 읽힌다.
  닉네임 자리는 공백 유지 (레이아웃 점프 방지).
- 성별 열이 먼저 차면: 해당 성별로 조인 시도 → RPC `GENDER_FULL` →
  "남자 자리가 모두 찼습니다" / "여자 자리가 모두 찼습니다" 스낵바.

### D.2 전체방 로비

현행 단일 그리드 유지. 빈 슬롯은 성별 미정이므로 FaIcon `user` (textHint, 18px) 를
1px border 원 안에.

### D.3 이성방 결과표 — 동성 쌍은 "안 보임" (마스킹 기각)

**결정: 남 × 여 직사각 매트릭스.** 8인 방 = 4행 × 4열 = 16셀. 동성 쌍은 계산도
저장도 표시도 하지 않는다.

- 행 = 보는 사람의 성별 쪽, 열 = 이성 쪽. "내 행 최상단 고정" 규칙이 그대로 성립
  (내가 여자면 여자들이 행). 셀 = 밴드 이모지, 탭 = 1🪙 unlock — 전부 현행 유지.
- "나와의 케미 순위" 리스트는 이성 참가자만 나열 (pairs 배열 순서 그대로).
- **검토 후 기각한 대안 — N×N 을 성별 블록 2×2 로 재배열하고 동성 블록 마스킹**:
  1. 8인 방에서 마스킹 면적이 화면의 약 43% (28쌍 중 12쌍) — 정보량 0 인 회색 벽이
     결과 화면의 절반을 차지한다.
  2. 마스킹 셀이 "잠긴 셀"로 읽혀 1🪙 unlock 셀과 혼동된다 — 돈 내면 열리는 줄 안다.
  3. 직사각 재배열은 셀 수가 절반 이하로 줄어 모바일에서 셀이 커진다 — 가독성 순이득.
- 베스트 쌍은 구조상 항상 이성 쌍 (이성 쌍만 계산하므로 별도 규칙 불필요).
- 엔진: `runBattle` 에 방 유형을 전달, 이성방이면 쌍 생성 단계에서 이성 쌍만 순회.
  동점 tie-break 규칙 (raw total → slot 사전순) 은 현행 그대로.

### D.4 아바타 규칙 — 썸네일 공개 옵션 (전 화면 공통 단일 규칙)

| 조건 | 로비·결과표·베스트 카드 아바타 |
|---|---|
| `thumb_open = true` | 얼굴 썸네일 (3단 fallback: 로컬 → CDN → 성별 아이콘) |
| `thumb_open = false` | male.png / female.png (참가자 성별 기준) |
| 베스트 매칭 당첨 2인의 상호 화면 | 항상 200×200 얼굴 사진 (§E.2 — thumb_open 무관) |

- 비공개방에서도 당첨 2인 화면만 예외 — 그 외 어떤 화면도 (당첨자 본인 화면 포함,
  제3자에게는) 사진을 보여주지 않는다.
- 웹 쇼케이스 (`/g/:id` completed): 사진 없음 정책 현행 유지 — thumb_open 이어도
  웹에는 안 나간다 (앱 내 로비 한정 공개).

## E. 베스트 매칭 성사 플로우

### E.1 사전 동의 — 조인 confirm 에서 계약한다

조인 confirm 시트 (공약 동의 문구를 대체):

> **이성방**: "베스트 매칭이 되면 상대에게 내 사진이 공개되고, 서로 동의하면 1:1
> 채팅이 열립니다" + [동의하고 참가] (PrimaryButton)
> **전체방**: "베스트 케미로 뽑히면 상대에게 내 사진이 공개됩니다" + [동의하고 참가]

참가 = 동의. 매칭 시점에 다시 사진 공개 여부를 묻지 않는다 (한쪽이 거부하면 반쪽
공개가 되는 어색한 상태를 만들지 않기 위해 입장 시점 일괄 동의로 설계).

### E.2 발표 시퀀스 (리빌 화면, 두 유형 공통)

```
3-2-1 카운트다운
  → 🏆 베스트 케미 카드 (전원)
  → [당첨 2인만] 매칭 사진 카드 + 채팅 동의 질문
  → 밴드 매트릭스 (전원)
```

**1) 베스트 케미 카드 (전원 동일)** — 두 사람의 아바타 (§D.4 규칙 — thumb_open
아니면 성별 아이콘), 닉네임, 점수, 밴드명. 현행 flip 연출 유지.

**2) 매칭 사진 카드 (당첨 2인 각자에게만)** — 베스트 카드 다음 장으로 flip-in:

> **eyebrow** (`caption` + gold): 베스트 매칭
> **타이틀** (`display`): 지은님과 매칭되었습니다
> **사진**: 상대의 200×200 얼굴 사진 (원형 아님 — 200×200 정사각, `AppRadius.xl`,
> 1px border). 아래 닉네임 `sectionTitle`.
> **각주** (`hint`): 이 사진은 매칭된 두 사람에게만 보입니다

**3) 채팅 동의 질문 (사진 카드 하단에 이어서)**:

> **질문** (`sectionTitle`): 지은님과 채팅방을 열까요?
> **보조** (`caption`): 두 사람 모두 열기를 선택하면 채팅방이 열립니다. 응답은
> 24시간 동안 가능하고, 선택은 되돌릴 수 없습니다.
> **액션**: [채팅방 열기] (PrimaryButton — 흰 bg + 1px textPrimary border) /
> [이번에는 넘어가기] (TextButton, textHint)

- "다음에" 류 카피 기각 — 응답이 1회 확정인데 다음 기회가 있는 것처럼 읽힌다.
  "이번에는 넘어가기"는 최종이면서 거절의 무게를 덜어낸다.

### E.3 응답 상태별 UX

응답 저장은 RPC `respond_match(team_id, accept)` (가드: status=completed, caller ∈
베스트 쌍, 24h 이내, 미응답 상태). 상태는 Realtime 구독 + 결과 화면 재진입 시 refetch.

| 상태 | 내 화면 | 상대 화면 |
|---|---|---|
| 나 수락, 상대 미응답 | 카드가 대기 상태로 전환 — "지은님의 선택을 기다리고 있습니다" (`body`) + `caption` "채팅방이 열리면 이 화면과 배틀 결과에서 들어갈 수 있습니다" | 질문 카드 그대로 (상대 수락 사실 비노출 — 압박 방지) |
| 둘 다 수락 | "채팅방이 열렸습니다" + [채팅 시작하기] (PrimaryButton) → 1:1 채팅 화면 | 동일 |
| 한쪽 거절 | (거절한 쪽) 질문 카드 접힘 — "채팅방을 열지 않았습니다. 결과표는 계속 볼 수 있습니다" | (상대 쪽) **즉시 종결** — "이번에는 채팅방이 열리지 않았습니다" + `caption` "결과표는 계속 볼 수 있습니다" |
| 24시간 무응답 | 양쪽 모두 — "응답 기간이 지나 채팅방이 열리지 않았습니다" | 동일 |

- **거절 즉시 종결 vs 24h 은닉 대기** 트레이드오프: 거절을 만료로 위장하면 완전
  은닉이 되지만 상대가 24시간 헛기다린다. 즉시 종결을 택한다 — 카피는 주어 없는
  결과 서술("열리지 않았습니다")로 통일해 누가 거절했는지 화면이 지목하지 않는다.
  시점으로 유추 가능한 것은 감수 (2인 관계에서 완전 은닉은 애초에 불가능).
- 거절·만료 결과 화면은 동일 위계 (danger 색 사용 금지 — 실패가 아니라 종결).
- 미당첨 참가자는 이 플로우를 전혀 보지 못한다 — 매칭 성사·미성사 여부도 제3자에게
  비노출 (베스트 카드까지만 공유 정보).

### E.4 채팅 진입과 재진입

- 성사 시 `chat_rooms` 행 생성 (양쪽 수락 트랜잭션 내). 1:1 채팅 화면: appBar =
  상대 닉네임 + 42px 아바타(사진), 본문 = 메시지 리스트 (`body`), 입력바 = TextField +
  전송 FaIcon. Supabase `chat_messages` insert + Realtime 구독 — 별도 설계 문서 없이
  이 구성으로 구현 가능한 최소 화면.
- 재진입 동선: ① 리빌 화면의 [채팅 시작하기], ② 내 배틀 목록의 완료 카드에 [채팅]
  outlined pill (당첨 + 성사 당사자에게만 노출).
- 배틀 30일 삭제 정책과 분리 — `chat_rooms.team_id` 는 on delete set null, 채팅은
  배틀 삭제 후에도 유지.

```
   ▼ 둘 다 수락 (24h 창)
1:1 채팅
```

### F.2 데이터 계약 변경 요약

| 대상 | 변경 |
