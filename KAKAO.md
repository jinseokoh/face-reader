# 카카오 초대하기 — 교감도 그룹 원격 참여

> 교감도(케미 그룹) 화면의 **[카카오톡으로 초대]** 버튼이 무엇을 하고, 초대받은
> 사람이 어떻게 원격으로 참여하는지, 그 전제·한계·장점을 쉽게 정리한 문서.
> (구현 단계: P3 — M1~M4 완료, M5·M6 미완. 아래 "한계" 참고.)

---

## 1. 한 줄 요약

방장이 만든 그룹을 **서버에 올리고**, 그 그룹의 주소(`facely.kr/g/{그룹ID}`)를
**카카오톡 메시지로 보낸다.** 받은 사람이 링크를 누르면 앱의 **참여 화면**이 열리고,
자기 얼굴을 등록하면 그 그룹의 멤버가 되어 케미 매트릭스에 합류한다.

---

## 2. 어떤 전제로 구현했나 (설계 결정)

1. **로컬 우선 + 필요할 때만 서버 (lazy sync)**
   그룹은 평소엔 폰 안(로컬)에만 산다. 현장에서 한 폰으로 여러 명을 등록하는
   "현장 경로"는 서버·로그인 없이 그대로 동작한다. **[카카오톡으로 초대]를 누르는
   그 순간 처음으로** 그룹이 서버에 올라간다. → 평소 마찰 0.

2. **원격 그룹의 주인은 "로그인한 계정"**
   여러 기기가 한 그룹을 공유하려면 "이 그룹의 방장이 누구인지" 흔들리지 않아야
   한다. 그래서 서버 그룹의 소유자는 익명이 아니라 **로그인 계정(auth.uid)**이다.
   초대를 누를 때 로그인이 안 돼 있으면 로그인 창을 먼저 띄운다.

3. **점수 계산은 앱이, 서버는 보관만**
   궁합 엔진은 앱(Flutter)에만 있다. 서버는 그룹·멤버 명단과 (마감 시) 결과표만
   저장한다. 멤버를 화면에 그릴 때 앱이 각 멤버의 측정값(metrics)을 서버에서
   읽어와 **그 자리에서 매트릭스를 계산**한다.

4. **링크 = 콘텐츠 (초대가 기능이 아니라 전제)**
   "우리 팀 궁합표"는 혼자 못 만든다. 링크 한 번이 곧 바이럴 루프의 시작점이다.

## 3. 원격 참여 과정 (단계별)

### 방장 쪽 — 초대 보내기
1. 그룹을 만든다(로컬). 멤버를 직접 등록하거나 대기 이름만 깔아둔다.
2. 그룹 화면에서 **[카카오톡으로 초대]** 탭.
3. **로그인 안 돼 있으면 로그인 창** → 로그인.
4. 앱이 그룹을 **서버에 push**: `teams` 1행(주인=나) + 이미 등록된 멤버들을
   `team_members`에 올린다. (대기 이름·아직 안 찍은 사람은 안 올라감.)
5. 카카오톡 공유 시트가 뜬다 — 메시지:
   > [그룹명] 관상학으로 풀어보는 우리 그룹내에서 나랑 가장 케미가 좋은 사람찾기에 참여해 보세요.
   `[참여하기]` 버튼 + 링크 `https://facely.kr/g/{그룹ID}`.
   (카톡 앱이 안 깔려 있으면 **OS 기본 공유 시트**(iOS·Android 시스템 공유 패널,
   `share_plus`)가 대신 뜬다 — 거기 나열된 문자(SMS)·메일·다른 메신저·"링크 복사"
   중에서 사용자가 직접 골라 "문구 + 링크" 텍스트를 보낸다. 카톡 전용 미리보기
   카드·[참여하기] 버튼은 이 경우 없음.)

### 초대받은 사람 쪽 — 참여하기 (앱 설치된 경우)
6. 카톡에서 링크를 누른다 → **유니버설/앱 링크**라 브라우저 대신 **앱이 열린다.**
7. 앱이 `/g/{그룹ID}` → **참여 화면(TeamJoinScreen)**으로 이동.
8. 화면이 서버에서 그룹을 읽어 **미리보기**를 보여준다 — 현재 참여자 칩, 그리고
   **방장이 미리 깔아둔 빈 자리(대기 이름) 칩** ("내 자리를 골라주세요").
9. 참여 방식 두 가지:
   - **(A) 내 자리 고르기** — 방장이 "까불이"로 깔아둔 빈 칩을 탭하면, 로그인 →
     (없으면) 셀카 등록 → **그 "까불이" 슬롯에 내 측정값이 채워진다(claim).**
     서버 키가 `(team_id, name)` 이라 같은 이름 행을 그대로 메운다.
   - **(B) 내 이름으로 새로 참여** — 하단 버튼. 빈 자리에 내가 없으면 내 닉네임으로
     새 멤버 추가.
   - 이미 누가 점유한 이름이거나 내가 이미 참여했으면 RLS가 막아 "이미 참여했거나
     사용 중인 이름이에요" 안내.
10. 그룹이 내 폰의 **"초대받은 그룹"** 목록에 들어오고, 그룹 화면으로 진입.

### 방장이 합류자를 보는 법
11. 그룹에 **다시 들어가거나**(자동 폴링), **홈 리스트·그룹 화면에서 당겨서
    새로고침**(pull-to-refresh)하면 앱이 서버를 다시 읽어(`refreshFromServer`)
    합류를 반영한다. 합류자가 "까불이" 슬롯을 채웠으면 방장 로컬의 그 대기 슬롯이
    **이름 매칭으로 채워지고**(중복 안 생김), 새 이름이면 멤버로 추가된다. 3명
    이상이면 부분 매트릭스가 보인다.

### 앱이 안 깔린 사람 — 웹 안내 페이지 (구현됨, M5)
- 링크를 누르면 브라우저로 `facely.kr/g/{id}` (`react/app/routes/g.$id.tsx`)가 열린다:
  - **마감 전** = 초대장(그룹명·"N명 참여 중"·"당신 자리가 비어 있어요"·참여자 칩
    + 앱 열기/스토어 CTA).
  - **마감 후** = 결과 쇼케이스(이름+밴드 이모지 매트릭스, 사진·점수 없음).
- 단 **설치 직후 자동 입장(deferred deep link)은 없다** — 설치 후 카톡에서 링크를
  다시 눌러야 그룹에 들어간다 (외부 SDK 필요라 의도적 제외, §5 참조).

## 5. 한계 · 단점

| # | 한계 | 영향 |
|---|---|---|
| 1 | **설치 후 자동 입장(deferred) 없음** | 웹 안내 페이지(`/g/{id}`)·스토어 유도는 **됨**(M5). 단 설치 직후 그 그룹으로 자동 진입은 안 되고, 카톡에서 링크를 다시 눌러야 함(외부 SDK 필요라 의도적 제외). |
| 2 | **실시간 아님 (폴링)** | 합류가 방장 화면에 즉시 안 뜬다. 그룹 재입장(자동 폴링) 또는 **당겨서 새로고침**(홈·그룹 화면)해야 반영. |
| 3 | **로그인 강제** | 익명 초대·익명 합류 불가. 로그인 단계가 마찰. |
| 4 | **이름 슬롯의 한계** | 합류자가 빈 슬롯("까불이")을 고르면 그 자리를 채운다(claim, 해결됨). 단 **이름이 키**라: 같은 이름 두 명은 한 슬롯만 가능, 점유된 이름은 못 가로챔, 방장이 합류 후 그 이름을 바꾸면 매칭이 깨질 수 있음. |
| 5 | **방장 대기 명단 원격 삭제 안 됨** | push 는 대기 이름을 insert-only(claim 보호)라, 방장이 로컬에서 대기 이름을 지워도 서버 슬롯은 남는다(다음 합류에 노출 가능). |
| 6 | **링크 = 누구나 읽기** | 그룹 read 권한이 "UUID 아는 사람"(링크 공유 모델). 링크가 외부로 새면 그룹 제목·멤버 이름·(옵트인) 썸네일이 노출된다. |
| 7 | **멤버마다 서버 왕복** | 매트릭스 그릴 때 앱이 멤버 수만큼 metrics를 개별 fetch(N회). 인원 많으면 로딩 지연. pull-to-refresh 도 그룹 수×멤버 수만큼 호출. |

## 6. 장점

- **현장은 무마찰, 원격은 필요할 때만** — 평소 로컬 우선이라 서버/로그인 없이 빠르게
  쓰고, 초대를 누를 때만 서버로 올라간다.
- **비파괴 동기화** — 방장이 그룹을 다시 push해도 이미 합류한 사람을 지우지 않는다
  (`(team_id, metrics_id)` unique upsert).
- **기존 인프라 재사용** — 측정값(metrics)·공유 수신(share_receive)·딥링크·AASA·
  카카오 SDK를 그대로 확장. 새 결제·새 서버 엔진 없음.
- **서버는 가볍다** — 궁합 엔진을 서버/웹에 포팅할 필요 없이, 앱이 계산하고 서버는
  명단·결과표(JSON)만 보관.
- **로컬 우선이라 견고** — 네트워크가 끊겨도 내가 만든 그룹은 폰에 그대로 있고,
  연결되면 동기화된다.
- **권한 분리** — DB RLS가 "방장만 명단 관리 / 합류자는 자기 자신만 추가"를 강제.
