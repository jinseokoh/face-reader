## 2. 두 트랙의 독립성 (이것이 모든 증상의 뿌리)

| | **Score Track (점수/막대)** | **Narrative Track (소제목 텍스트)** |
|---|---|---|
| **입력** | 메트릭 raw 값 (double) | 메트릭 정수 버킷 (int) |
| **처리** | 가중합 → 원점수 → v9 normalize | 정수 비교 조건 (`>= 1`, `<= -1`) AND 매칭 |
| **출력 단위** | 10개 attribute 모두 점수 보장 | Trigger된 rule만 텍스트 블록 |
| **보장** | v9가 수학적으로 최소 3.0 spread, 최상위 ≥ 8.0 | **보장 없음**. 평범한 얼굴은 rule이 하나도 발동 안 할 수도 있음 |
| **UI** | 막대 차트에 10개 attribute가 항상 다 표시됨 | attribute별로 rule이 0개면 그 소제목은 **아예 존재하지 않음** |

> **중요**: Rule 조건은 정수 버킷이 `>= 1`(평균에서 1 표준편차 이상) 같은
> **꽤 보수적인 컷오프**를 사용합니다. 얼굴이 모든 메트릭에서 평균 근처면
> 아무 rule도 걸리지 않아 **내러티브가 사실상 비어버릴 수 있습니다.**

## 3. Rule은 언제 발동하는가 (report_assembler의 착각을 풀기)

`attribute_engine.dart:221-318`을 보면 common rule 50개 + gender rule 10개 + age rule 5개가 있습니다. 전형적인 조건 예:

```dart
_InteractionRule('L-R1',
  (s) => s['gonialAngle']! >= 1 && s['eyeCanthalTilt']! >= 1, …)
```

즉 "턱 각도가 평균보다 1 이상 강하고 **동시에** 눈꼬리 기울기가 1 이상"이어야 L-R1이 발동합니다. 두 조건 **AND**입니다.

- 한 attribute당 common rule 5개 × 조건은 거의 모두 AND로 묶인 2-메트릭 조합
- 얼굴의 특성이 "어느 쪽도 극단적이지 않다"면 어떤 rule도 발동 안 함
- 결과: **해당 attribute의 소제목 자체가 조립되지 않음**

`report_assembler.dart`의 흐름 (현재 코드):

```dart
// 1. triggered된 rule id로 text block 모으기
final matchedBlocks = …where(triggered).toList();

// 2. block을 attribute로 묶기
final blocksByAttribute = …;  // ★ rule이 하나도 발동 안 한 attribute는 여기에 없음

// 3. 이 Map의 key만 점수순 정렬 후 top 5
final topAttributes = sortedAttributes.take(5).toList();
```

여기서 `.take(5)`는 사실 **원인이 아닙니다**. `blocksByAttribute`에 이미 2개만 들어있으면 top 5를 취해도 2개뿐입니다. **진짜 원인은 나머지 8개 attribute의 rule이 하나도 발동하지 않아 그 소제목이 시작부터 존재하지 않는다는 것**입니다.

## 4. v9 calibration이 실제로 바꾼 것 / 바꾸지 않은 것

### v9가 **바꾼 것** (점수/막대 경로만)
`attribute_engine.dart:409-528`, `normalizeAllScores()`:

1. **입력**: 10개 attribute의 raw 원점수 전체 (이전엔 attribute 하나씩 독립 처리)
2. **처리**:
   - 각 attribute의 global percentile 계산 (calibration CDF 사용)
   - 같은 얼굴의 10개 percentile을 내림차순 정렬 → within-face rank
   - **60% rank + 40% global percentile**로 혼합
   - 5.0 ~ 10.0으로 매핑
3. **보장**:
   - 같은 얼굴 내 최소 spread 3.0 이상
   - 최상위 attribute ≥ 8.0
   - 최하위 attribute ≤ 7.0
4. **동기**: v8은 "모든 유저의 top 3가 동일"하게 saturate되는 문제가 있었음

### v9가 **바꾸지 않은 것** (내러티브 경로)
- `evaluateRules()` 함수 (line 375)는 `Map<String, int> scores`를 입력으로 받음
- 이 `scores`는 **메트릭 z-score 버킷**이지 **attribute 점수가 아님**
- Rule 조건은 `s['gonialAngle']! >= 1` 처럼 **메트릭 버킷만 체크**
- v9가 바꾼 것은 **attribute 원점수 → 표시 점수**뿐
- 따라서 **v9는 rule 발동 개수에 이론적으로 영향을 미치지 않습니다**

### 그럼 왜 "엔진 업그레이드 후 짧아졌다"고 느끼는가?

v9 자체가 rule 개수를 줄이진 않았지만, **인지적 대비**를 키웠습니다:

- v8 시절: 점수 분포가 좁아서 모든 attribute가 6~7점대로 다닥다닥 붙어있음 → 유저가 "왜 텍스트는 2개뿐인가"에 둔감
- v9 이후: 점수 spread 보장으로 **최상위 attribute가 8, 9, 10점으로 선명하게** 보임 → "9점인 지능 얘긴 어디 갔나?"라는 불일치가 **시각적으로 도드라짐**
- **동일한 수의 rule이 발동하더라도, 유저 입장에서는 "막대는 강렬한데 글은 비어있다"라는 격차가 크게 느껴짐**

즉 v9는 증상의 **원인이 아니라 증상의 가시성 증폭기**입니다.
원래부터 존재하던 "평범한 얼굴은 rule 발동이 희박하다"는 구조적 한계가 v9의 강한 점수 대비 때문에 눈에 띄게 된 것입니다.

## 5. 해결 방향 — Combinatorial Rule Expansion

### 5.1 먼저, 폐기된 오답들

이 문서는 처음에 세 가지 "해결책"을 거쳐서 여기 도달했다. 전부 틀렸고, 왜 틀렸는지 기록해 둔다. **전부 state space를 축소하는 방향이었기 때문**에 관상 앱의 본질과 정면 충돌한다.

| 폐기된 아이디어 | 왜 틀렸나 |
|---|---|
| **Fallback 텍스트 (score band 5개)** | 평범한 얼굴 전부가 같은 "기본 설명"을 받게 됨. 10 attribute × 5 band = **50개 텍스트로 세상 모든 얼굴을 커버**하는 꼴. variation 완전 붕괴. |
| **같은 룰의 tier 3x 복제 (strong/medium/soft)** | 같은 텍스트를 세 가지 톤으로 물 타는 것. 룰 **개수**는 늘지만 unique content는 그대로. 그냥 유지보수 지옥. |
| **Score-driven narrative (룰 폐기)** | 전통 관상학의 조합 해석이 전부 증발. "리더십 8.3점"이라는 숫자가 왜 그 점수인지, 어떤 조합이 만들었는지 말할 수단이 사라짐. 궁합 엔진도 단조로워짐. |

**공통 오류**: "룰이 발동 안 하면 다른 걸로 메꾸자"는 프레임. 진짜 질문은 **왜 룰이 발동 안 하는가**였다.

### 5.2 진짜 진단 — 룰 공간이 한쪽 구석에만 몰려 있다

`attribute_engine.dart`의 현재 상태를 수치로 보자:

- **메트릭 12개 × 2-metric 페어**: 중복 제거 후 **66 페어** (정면만 기준)
- **각 페어의 의미 있는 threshold 패턴**: 대략 12~18개 (예: `++`, `+0`, `+-`, `0+`, `00`, `0-`, `-+`, `-0`, `--`, `+2`, `-2`, `|·|≤1` 등)
- **이론적 2-metric 룰 공간 (정면만)**: 약 **800~1,200 슬롯**
- **현재 커버**: common 38 + gender 10 + age 5 = **53개 룰 (약 5~7%)**
- **공간 분포**: 거의 전부 `A>=1 && B>=1` **대각선 한 구석**에 집중

### 5.2.1 Phase 0.5 (측면 캡처) 이후 — 룰 공간이 4배로 확장됨

2026-04-14에 측면(3/4 view) 캡처를 도입하면서 메트릭 공간이 크게 넓어졌다. 카메라와 앨범 둘 다 **정면 + 선택적 측면 사진**을 받을 수 있으며, 측면에서는 사상면(sagittal) 구조를 측정하는 **7개 z-score 메트릭 + 2개 binary flag**가 추가됐다:

| 측면 메트릭 | 단위 | 측정 대상 |
|---|---|---|
| `nasofrontalAngle` | ° | 비전두각 (이마-코 경사) |
| `nasolabialAngle` | ° | 비순각 (코끝 들림/처짐) |
| `facialConvexity` | ° | 안면 돌출각 (G-Sn-Pog) |
| `upperLipEline` | norm. | 상순 E-line 거리 (입술 돌출) |
| `lowerLipEline` | norm. | 하순 E-line 거리 |
| `mentolabialAngle` | ° | 순이각 (입술-턱 경계 깊이) |
| `noseTipProjection` | ratio | 코끝 돌출 비율 |
| **flag** | — | — |
| `aquilineNose` | bool | **매부리코** (rhinion Z-displacement detection) |
| `snubNose` | bool | 들창코 (nasolabialAngle > 110°) |

**새 메트릭 공간 크기**:
- 17 정면 메트릭 + 7 측면 메트릭 = **24 연속 메트릭** → 2-metric 페어 **~180개**
- Binary flag 2개는 모든 연속 메트릭과 조합 가능 → **~48 additional 조합**
- **이론적 총 룰 공간: ~3,500-5,000 슬롯** (이전 800-1,200의 4-5배)

**실무적 의미**:
- 매부리코는 이제 **진짜 측정 가능한 메트릭**. Proxy가 아닌 직접 flag.
- "강한 턱 + 매부리코"(강인형 지도자), "처진 코끝 + 얇은 입술"(침착형), "안면 돌출 + 짧은 인중"(정력형) 같은 전통 관상학의 조합 해석이 이제 룰로 표현 가능.
- 측면 데이터가 없는 얼굴(앨범에서 1장만, 혹은 카메라에서 건너뛰기)은 측면 관련 룰이 자동으로 **skip** — degraded mode로 깔끔하게 동작.
- `lateralScores == null`일 때 `_lateralRules` 리스트 전체가 평가되지 않으므로 lateral-only 룰이 정면만 있는 사용자에게 잘못 발동할 가능성 zero.

#### 현재 룰의 패턴별 분포 (common 38개 기준, 대략)

| 패턴 | 의미 | 개수 | 트리거 인구 |
|---|---|---|---|
| `A>=1 && B>=1` | 두 메트릭 모두 z≥0.5 (**같이 튐**) | 22 | ~9% |
| `A>=2 && B>=1` | 한쪽 극단 + 다른 쪽 강함 | 1 | ~1% |
| `A<=-1 && B<=-1` | 둘 다 약함 | 4 | ~9% |
| `A>=1 && B<=-1` | **대조 패턴** (contrast) | 4 | ~9% (비어 있다시피) |
| `A>=1 && B>=0` | 한쪽 튐 + 다른 쪽 중립 이상 | 2 | ~25% |
| `\|A\|<=1 && \|B\|<=1` | **중용 패턴** (balanced) | 1 | ~40% |
| `A==0 && B==0` | 둘 다 평균 근처 | 0 | ~50% |
| 단일 메트릭 | 한쪽만 고려 | 0 (gender/age엔 약간) | 다양 |

**이 표가 핵심이다.** "평범한 얼굴"의 거의 전부가 떨어지는 `중용`, `한쪽만 튐`, `평균 근처`, `대조` 영역에 **룰이 거의 존재하지 않는다**. 그러니 평범한 얼굴은 룰이 "안 뚫리는" 게 아니라 **그 영역에 뚫을 룰 자체가 없다**. v9는 이 빈 공간이 얼마나 큰지 막대 차트로 시각화했을 뿐 원인이 아니다.

### 5.3 해법 — 룰 공간을 densely populate

**목표**: 62 → **300~500개** 룰. 각 룰은 **고유한 관상학 해석 텍스트**를 가진다 (약한 버전, 복제 아님).

**원칙**:

1. **모든 의미 있는 2-metric 페어에 multi-band 룰 배치**
   - `A>=2 && B>=1` — 극적 조합 (드물고 인상적)
   - `A>=1 && B>=1` — 강한 결합 (현재 주종)
   - `A>=1 && B==0` — 한쪽만 튐 (단일 특성 부각)
   - `A>=1 && B<=-1` — **대조 패턴** (외유내강, 유연과 강건의 공존 등)
   - `|A|<=1 && |B|<=1` — **중용 패턴** (균형의 미학)
   - `A==0 && B>=1` — 평균에 가까운 특성 + 한쪽 튐
   - `A<=-1 && B<=-1` — 둘 다 약함 (취약 조합)
   - 가능한 모든 의미 있는 sign × magnitude 조합

2. **각 패턴은 고유 텍스트**
   - `gonialAngle>=1 && eyeCanthalTilt>=1` → "강단과 날카로움의 결합, 전형적 리더형"
   - `gonialAngle>=1 && eyeCanthalTilt<=-1` → "강한 턱에 순한 눈매, 외유내강의 전형"
   - `|gonialAngle|<=1 && |eyeCanthalTilt|<=1` → "균형 잡힌 이목구비, 중도를 걷는 처신"
   - `gonialAngle==0 && eyeCanthalTilt>=2` → "평이한 턱선에 유달리 날카로운 눈, 관찰형"
   - **셋 다 물 타기가 아니라 서로 다른 관상학적 해석**이다. 문헌에는 이런 대조·중용 해석이 원래부터 가득하다.

3. **단일 메트릭 룰도 소수 허용**
   - 예: `lipFullnessRatio >= 2` → "유난히 풍성한 입술, 정열적 표정"
   - 현재 common 룰엔 없음 (gender 룰에만 하나). 단일 메트릭 룰은 **variation을 희생하지 않으면서 coverage를 늘리는 값싼 방법**이다. 단, 전체의 10%를 넘지 않게 제한 (조합의 맛이 죽지 않도록).

4. **작은 input 변화 → 완전히 다른 글**
   - 한 메트릭의 bucket이 0 → 1로 바뀌면 **그 메트릭이 참여하는 모든 룰의 발동 상태가 재편성**된다.
   - 12 메트릭 중 5개의 bucket이 다르면 → 수십 개 룰이 다른 set으로 발동 → **매 얼굴마다 고유한 텍스트 몽타주**
   - 이것이 유저가 기대하는 "얼굴 바꿀 때마다 다른 얘기 나오는" 감각이다.

### 5.4 엔진 변경 범위 (놀라울 정도로 작다)

#### Phase 1 (룰 densification)에서 손대는 곳
| 파일 | 변경 |

## 6. 요약 한 문단

**"신뢰성/바람기만 보이는 현상"의 뿌리는 v9도, 룰 엔진 구조도, threshold 높이도 아니다. 룰 공간이 한쪽 구석에만 몰려 있다는 것이다.** 현재 ~53개 common/gender/age 룰의 대부분이 `A>=1 && B>=1` 대각선 패턴에 집중되어 있고, 이게 통계적으로 상위 ~9% 얼굴만 걸러낸다. 평범한 얼굴이 주로 떨어지는 중용 패턴(`|A|≤1 && |B|≤1`), 대조 패턴(`A>=1 && B<=-1`), 한쪽만 튐 패턴(`A>=1 && B==0`) 영역에는 룰이 **거의 존재하지 않는다**. 즉 룰이 "안 뚫리는" 게 아니라 **뚫을 룰 자체가 그 영역에 없다**. v9는 점수 spread를 명확히 해서 이 빈 공간을 시각적으로 드러냈을 뿐 원인은 아니다. 해결은 룰 공간을 densely populate 하는 것 — 62 → 300~500개로 확장하되 각 룰이 **고유한 관상학 해석 텍스트**를 가지도록 해서 작은 input 차이에도 전혀 다른 글이 나오게 한다. 이 방향은 variation을 **증가**시킨다. 초기에 검토한 fallback 텍스트·tier 복제·score-driven narrative는 전부 variation을 축소하는 방향이라 폐기. v9, 궁합 엔진, report_assembler는 무변경이고, 변경 범위는 `_commonRules` 리스트와 `rule_text_blocks.dart`에 항목을 **추가**하는 것뿐이다.

## 7. 쉬운 설명 (중학생 버전)

### 비유: 요리책이 너무 얇다

우리 앱 안에는 두 명의 선생님이 있어요.

**점수 선생님 (막대 차트 담당)**
- 10개 과목(지능, 신뢰성, 바람기, 인간관계, 재물운 등)마다 **무조건 점수를 매깁니다.**
- 그래서 막대 그래프에는 언제나 10개 막대가 다 나와요.

**글쓰기 선생님 (분석 문장 담당)**
- 이 선생님은 **"요리책(레시피 목록)"**을 보고 글을 씁니다.
- 레시피는 이런 식이에요: **"재료 A가 많이 있고 + 재료 B도 많이 있으면 → '리더십' 문단을 써라"**
- 재료란 얼굴의 메트릭(턱 각도, 눈 기울기, 입술 두께…)이에요.
- 그런데 **요리책에 레시피가 겨우 ~60개**뿐입니다.
- 그리고 그 레시피의 거의 전부가 **"재료 A 많이 + 재료 B 많이"** (둘 다 극단적) 패턴이에요.

### 왜 평범한 얼굴엔 글이 거의 안 나오는가?

대부분 사람의 얼굴을 재료로 따지면 이런 식입니다:
- "재료 A는 보통, 재료 B는 조금 많음, 재료 C는 조금 적음…"
- 극단적으로 많은 재료가 거의 없어요.

그런데 요리책의 레시피는 **"둘 다 극단적이어야"** 발동해요. 그래서 대부분 사람에게는 레시피가 **하나도 작동하지 않습니다.** 그 과목(예: 지능)의 글이 텅 비어 보이는 이유예요.

**중요한 오해 풀기**: 이건 "레시피의 기준이 너무 높아서" 생긴 문제가 **아닙니다.** 기준을 낮춰서 "조금만 많아도 발동"하게 만들면 **모든 사람이 똑같은 레시피**를 받게 돼요. 얼굴마다 다른 이야기가 안 나옵니다. 재미가 없어져요.

**진짜 문제는 레시피의 "종류"가 너무 적다는 것입니다.**

지금 요리책에는 거의 "둘 다 많이" 레시피만 있어요. 있어야 할 건:
- "재료 A 많이 + 재료 B **적음**" — 대조 레시피 (예: 외유내강)
- "재료 A, B 둘 다 **보통**" — 중용 레시피 (균형의 미학)
- "재료 A 많이 + 재료 B 보통" — 한쪽만 튐 레시피
- "재료 A 보통 + 재료 B 많이" — 반대도 마찬가지
- 극단 조합부터 완전히 평범한 조합까지, 다 저마다 고유한 해석이 있어야 해요.

전통 관상학 책을 펼치면 이런 해석이 **원래 가득**합니다. 앱의 요리책이 그걸 거의 못 담고 있는 거예요.

### v9 업데이트는 뭘 했나?

우리 앱은 얼마 전 "v9 calibration"이라는 업데이트를 했어요. 한 일은 딱 하나:
- **점수 선생님의 점수 매기는 방식을 더 똑똑하게** 만든 것.
- 예전엔 10개 과목이 다 6~7점대에 다닥다닥 붙어 있어서 비슷해 보였어요.
- 이제는 가장 잘하는 과목은 8점 이상, 가장 못하는 과목은 7점 이하로 **점수 차이가 뚜렷**해요.

**글쓰기 선생님은 전혀 건드리지 않았습니다.** 요리책도, 레시피 조건도 그대로.

### 그런데 왜 유저는 "분석이 짧아졌다"고 느끼는가?

- v9 전: 모든 과목 점수가 고만고만해서 글과 막대의 불일치를 잘 못 느꼈어요.
- v9 후: 막대가 뚜렷해지니까 "9점짜리 과목 있는데 글이 왜 없지?"라는 의문이 **눈에 띄게 커진 것**이에요.
- **실제로 글이 짧아진 게 아니라, 짧은 게 티가 나게 된 것.**

### 어떻게 고칠 것인가 — 요리책을 두껍게

v9 점수 매기는 방식은 **건드리지 않습니다.** 그 대신 **요리책에 레시피를 잔뜩 추가**합니다:

- 지금: ~60개
- 목표: **300~500개**
- 모든 레시피는 **고유한 관상학 해석**을 가집니다. "약한 버전 복사"가 아니에요.

그러면 이런 일이 벌어져요:
- 어떤 얼굴이든 **여러 레시피가 동시에 발동**해서 글이 풍성해집니다. 평범한 얼굴도 대조·중용·한쪽만 튐 레시피가 몇 개씩 걸려요.
- 재료(메트릭)가 한 칸만 달라져도 **발동되는 레시피 set이 재편성**되어 완전히 다른 글이 나옵니다.
- 얼굴 A와 얼굴 B를 궁합으로 비교할 때도 **매번 다른 해석**이 나옵니다.

**결과**: "다양성이 늘어난다." 관상 앱의 재미가 바로 이 다양성이에요.

### 우리가 한 번 잘못된 길로 갔던 이야기

처음엔 "레시피 발동 안 한 과목엔 점수 보고 기본 설명 써주자"(fallback) 같은 생각을 했어요. **그런데 이건 바보 같은 답**이었습니다. 모든 평범한 얼굴이 똑같은 기본 설명을 받게 돼서, 앱의 재미(다양성)가 완전히 사라지거든요. "tier 여러 개로 복제하자"도 마찬가지, "점수만 보고 글 쓰자"도 마찬가지 — 전부 **경우의 수를 줄이는 방향**이라 틀렸어요.

**진짜 답은 경우의 수를 늘리는 것.** 요리책을 5~8배 두껍게 만들기. 이게 관상학의 본질(조합 해석)과 앱의 본질(다양성)을 **둘 다** 살리는 유일한 길입니다.

### 한 줄 정리

막대는 10개 다 보이는데 글은 2개만 나오는 이상함은 **"레시피 기준이 너무 높아서"가 아니라 "요리책에 레시피 종류가 너무 적어서"**입니다. 해결은 **"요리책을 두껍게" — 60개 → 300~500개로 레시피를 늘리되 각 레시피에 고유한 해석을 담는 것**입니다.

## 8. 왜 점수 트랙과 내러티브 트랙을 굳이 2원화했는가

"둘로 나뉘어 있는 것 자체가 문제 아닌가? 하나로 합치면 안 되나?"라는 의문이 자연스럽게 들 수 있습니다. 답: **2원화는 버그가 아니라 의도적 디자인**입니다. 두 트랙은 각자 **다른 종류의 질문에 답하기 위해** 태어났고, 서로 겹치는 것처럼 보여도 실제로는 답하는 영역이 다릅니다.

### 8.1 점수 트랙이 답하는 질문
- "이 얼굴의 10개 특성을 **한눈에 비교**하고 싶다"
- "**다른 사람과 비교**하려면 숫자가 필요하다" — 궁합 엔진(`compatibility_engine.dart`)이 얼굴 A와 B의 attribute 벡터를 비교해서 궁합 점수를 계산합니다. 이때 "rule이 발동했느냐"로는 수치 비교가 불가능합니다.
- "UI에 **막대 그래프**를 그리려면 연속값이 필요하다"
- 요구 성질: **빠짐 없음, 비교 가능, 연속적(continuous)**

→ 그래서 **메트릭 가중합 → v9 정규화 → 5.0~10.0 연속 점수**라는 형태가 자연스럽게 선택됐습니다. 모든 얼굴의 모든 attribute가 항상 숫자를 가지게 되는 것이 이 트랙의 본질입니다.

### 8.2 내러티브 트랙이 답하는 질문
- "이 얼굴이 **어떻게 특별한가**를 한국어 관상학 문장으로 설명하고 싶다"
- 전통 관상학의 해석은 **단일 특징이 아니라 조합에서 나옵니다** — "턱이 강하면서 동시에 눈매가 날카로우면 … 한다"와 같이 두 메트릭 이상의 **AND 결합 패턴**이 의미를 만듭니다.
- Rule 조건이 `gonialAngle >= 1 && eyeCanthalTilt >= 1`처럼 AND로 묶인 이유가 바로 이것입니다.
- 점수로는 "리더십 8.3점"이라고 말할 수 있지만, 그 점수를 **왜** 받았는지, **어떤 조합 때문인지**는 점수만으로는 표현할 수 없습니다.
- 요구 성질: **구체적, 패턴 기반, 조건이 맞았을 때만 의미 있음(discrete)**

→ 그래서 **메트릭 버킷 조건 → triggered rule → text block → 조립**이라는 형태가 선택됐습니다. 이 트랙의 본질은 **특별한 조합이 있을 때만 무언가 말한다**는 것입니다.

### 8.3 왜 합치면 안 되는가 — 합치는 세 가지 방법과 그 대가

| 합치는 방법 | 잃는 것 |
|---|---|
| **① 점수만으로 텍스트 자동 생성 (rule 제거)** | 관상학의 조합 해석이 통째로 사라짐. "당신의 지능은 8.3점입니다" 식의 로봇 톤 생성문만 남아 전통 관상학의 맛이 증발. 또한 "어느 조합이 이 점수를 만들었는가"가 말해지지 못함 |
| **② Rule만으로 차트 대체 (점수 제거)** | 평범한 얼굴은 rule이 0~2개 발동 → 차트에 막대 1~2개만 존재. 10개 특성 비교 UI 불가능. 궁합 엔진이 점수 벡터를 요구하므로 **궁합 기능 자체가 붕괴** |
| **③ 점수를 rule 트리거로 전용 (score ≥ 8 → rule 발동)** | rule의 장점인 "두 메트릭 조합 해석"을 포기해야 함. 점수가 낮아도 의미 있는 조합(예: "약한 턱 + 얇은 입술" 같은)이 있을 수 있는데 그것들이 모조리 소실됨. 또한 v9의 rank-aware 정규화는 최상위 attribute가 항상 8 이상이므로 결국 **상위 1~2개만 rule이 붙게 됨** — 현재 증상이 재현 |

### 8.4 두 트랙의 관계 — 표로 정리

| | 점수 트랙 | 내러티브 트랙 |
|---|---|---|
| **입력** | 메트릭 raw 값 (double) | 메트릭 정수 버킷 (int) |
| **수학** | 가중합 → percentile → rank-blend → 5~10 | 정수 비교 AND 체인 |
| **출력 단위** | attribute 10개 모두 (빠짐 없음) | 발동된 rule만 (희박할 수 있음) |
| **답하는 질문** | "이 얼굴은 10개 특성 중 어디가 강하고 약한가?" | "이 얼굴의 어떤 조합이 특별한가?" |
| **UI 용도** | 막대 차트, 궁합 엔진 | 한국어 서술문 |
| **성질** | 정량, 연속, 비교 가능 | 정성, 이산, 패턴 기반 |
| **v9 영향** | 받음 (정규화 방식 변경) | 받지 않음 (rule 엔진은 독립) |

두 트랙은 **동일한 12개 메트릭에서 갈라져 나와** 서로 독립적으로 결과를 만들어냅니다. 합치지 않은 이유는 각자가 잘하는 일이 본질적으로 다르기 때문입니다: **"비교 가능한 숫자"**와 **"패턴 기반 해석"**은 서로 다른 정보 형태이고, 각자 잘하는 걸 시키는 게 현재 구조입니다.

### 8.5 내러티브 트랙은 왜 반드시 combinatorial이어야 하는가

§8.1~8.4는 "두 트랙을 **합치면 안 된다**"를 설명했다. 한 걸음 더 나아가, **내러티브 트랙 내부에서도** "룰 제거 → score 기반 생성" 같은 단순화를 하면 안 된다. 이유는 세 가지다.

**(1) Variation이 상품이다**

관상 앱의 유저가 사는 것은 **"내 얼굴을 찍으면 매번 다른 얘기가 나온다"**는 경험이다. 친구와 비교해도 다르고, 다음 달에 다시 찍어도 다르고, 궁합 상대가 달라져도 다르게 나와야 재미있다. 이 재미의 원천은 **입력 공간의 작은 변화가 출력 공간의 큰 변화로 증폭되는 것**이다.

점수 기반 생성(점수 → 5개 band → 고정 문장)은 정반대로 간다: **점수가 5.2든 5.7이든 같은 band에 떨어지면 같은 글이 나온다**. 얼굴 A와 얼굴 B가 메트릭 분포가 완전히 달라도 v9 점수가 비슷하면 텍스트는 동일해진다. 이건 앱의 존재 이유 자체를 무너뜨린다.

조합 룰은 반대다: 얼굴 A와 B가 점수가 같아도 메트릭 bucket이 다르면 발동하는 룰 set이 전혀 다르고, 따라서 글의 몽타주가 완전히 달라진다. **input space의 차이를 narrative space로 팽창시키는 메커니즘이 바로 룰**이다.

**(2) 궁합의 재미는 조합의 재미다**

유저가 궁합을 보는 이유는 "A와 B의 조합이 왜 특별한가"이다. 이때 A의 "강한 턱 + 순한 눈매"와 B의 "약한 턱 + 날카로운 눈매"가 만나는 서사는 **조합 룰 패턴의 교집합 / 차집합**으로만 표현 가능하다. 점수 벡터 cosine similarity로는 "외유내강과 내유외강의 보완"같은 해석이 나오지 않는다.

궁합 엔진이 재미있어지려면 **양쪽 얼굴 모두 풍부한 룰 set**을 가져야 한다. 한쪽이 비어 있으면 비교할 게 없다. 이게 §5의 densification이 궁합 기능에도 직접적으로 기여하는 이유다.

**(3) 전통 관상학은 본질적으로 조합 기반이다**

"턱이 강하면서 입꼬리가 올라가면 X", "눈썹이 굵은데 인중이 짧으면 Y", "균형 잡힌 이목구비에 유난히 깊은 눈은 Z" — 관상학 문헌의 해석은 전부 2~3개 메트릭의 AND/대비 조건에서 파생된다. 이 장르의 언어를 "점수 하나"로 압축하면 관상학이라는 도메인 자체가 소실된다. **우리가 만드는 것은 점수 매기는 계산기가 아니라 관상학 엔진**이다.

따라서 **룰 엔진은 반드시 살려야 하고, 문제는 룰의 개수와 분포**라는 결론이 여기서 자연스럽게 나온다.

### 8.6 올바른 해법은 densification — 한 문단 요약

점수 트랙과 내러티브 트랙이 분리되어 있는 이유는 §8.1~8.4에서 설명했다: 한쪽은 "빠짐없는 비교 가능한 숫자", 다른 쪽은 "패턴 기반 조합 해석"이라는 **원리적으로 다른 성질**의 정보를 만들기 때문이다. 이 분리 구조는 유지된다. 문제가 발생한 지점은 분리 구조 자체가 아니라 **내러티브 트랙 안의 룰 공간이 한쪽 구석(`A>=1 && B>=1`)에만 몰려 있다**는 것이다. 평범한 얼굴이 주로 떨어지는 중용·대조·asymmetric 영역이 거의 비어 있어서 글이 안 나온다. 해결은 두 트랙을 합치거나 점수에서 텍스트를 빌려오는 것이 **아니라**, 내러티브 트랙의 룰 공간을 **원래 커버했어야 할 크기로 densely populate** 하는 것이다. 이 방향은 variation을 줄이지 않고 오히려 5~8배로 키운다. 관상학의 조합 전통, 궁합 엔진의 풍부함, 유저가 기대하는 "매번 다른 얘기"가 모두 이 densification으로 동시에 해결된다.
