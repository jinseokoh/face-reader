# 관상 분석 엔진 Big Picture

이 문서는 "분석 결과가 왜 신뢰성/바람기 두 개만 나오는가?"와
"v9 calibration이 어디에 영향을 주었고, 어디에는 주지 않았는가?"를
한눈에 이해하기 위한 구조 설명서입니다.

## 2. 두 트랙의 독립성 (이것이 모든 증상의 뿌리)

| | **Score Track (점수/막대)** | **Narrative Track (소제목 텍스트)** |
|---|---|---|
| **입력** | 메트릭 raw 값 (double) | 메트릭 정수 버킷 (int) |
| **처리** | 가중합 → 원점수 → v9 normalize | 정수 비교 조건 (`>= 1`, `<= -1`) AND 매칭 |
| **출력 단위** | 10개 attribute 모두 점수 보장 | Trigger된 rule만 텍스트 블록 |
| **보장** | v9가 수학적으로 최소 3.0 spread, 최상위 ≥ 8.0 | **보장 없음**. 평범한 얼굴은 rule이 하나도 발동 안 할 수도 있음 |
| **UI** | 막대 차트에 10개 attribute가 항상 다 표시됨 | attribute별로 rule이 0개면 그 소제목은 **아예 존재하지 않음** |

> **중요**: Rule 조건은 정수 버킷이 `>= 1`(평균에서 1 표준편차 이상) 같은
> **꽤 보수적인 컷오프**를 사용합니다. 얼굴이 모든 메트릭에서 평균 근처면
> 아무 rule도 걸리지 않아 **내러티브가 사실상 비어버릴 수 있습니다.**

## 3. Rule은 언제 발동하는가 (report_assembler의 착각을 풀기)

`attribute_engine.dart:221-318`을 보면 common rule 50개 + gender rule 10개 + age rule 5개가 있습니다. 전형적인 조건 예:

```dart
_InteractionRule('L-R1',
  (s) => s['gonialAngle']! >= 1 && s['eyeCanthalTilt']! >= 1, …)
```

즉 "턱 각도가 평균보다 1 이상 강하고 **동시에** 눈꼬리 기울기가 1 이상"이어야 L-R1이 발동합니다. 두 조건 **AND**입니다.

- 한 attribute당 common rule 5개 × 조건은 거의 모두 AND로 묶인 2-메트릭 조합
- 얼굴의 특성이 "어느 쪽도 극단적이지 않다"면 어떤 rule도 발동 안 함
- 결과: **해당 attribute의 소제목 자체가 조립되지 않음**

`report_assembler.dart`의 흐름 (현재 코드):

```dart
// 1. triggered된 rule id로 text block 모으기
final matchedBlocks = …where(triggered).toList();

// 2. block을 attribute로 묶기
final blocksByAttribute = …;  // ★ rule이 하나도 발동 안 한 attribute는 여기에 없음

// 3. 이 Map의 key만 점수순 정렬 후 top 5
final topAttributes = sortedAttributes.take(5).toList();
```

여기서 `.take(5)`는 사실 **원인이 아닙니다**. `blocksByAttribute`에 이미 2개만 들어있으면 top 5를 취해도 2개뿐입니다. **진짜 원인은 나머지 8개 attribute의 rule이 하나도 발동하지 않아 그 소제목이 시작부터 존재하지 않는다는 것**입니다.

## 4. v9 calibration이 실제로 바꾼 것 / 바꾸지 않은 것

### v9가 **바꾼 것** (점수/막대 경로만)
`attribute_engine.dart:409-528`, `normalizeAllScores()`:

1. **입력**: 10개 attribute의 raw 원점수 전체 (이전엔 attribute 하나씩 독립 처리)
2. **처리**:
   - 각 attribute의 global percentile 계산 (calibration CDF 사용)
   - 같은 얼굴의 10개 percentile을 내림차순 정렬 → within-face rank
   - **60% rank + 40% global percentile**로 혼합
   - 5.0 ~ 10.0으로 매핑
3. **보장**:
   - 같은 얼굴 내 최소 spread 3.0 이상
   - 최상위 attribute ≥ 8.0
   - 최하위 attribute ≤ 7.0
4. **동기**: v8은 "모든 유저의 top 3가 동일"하게 saturate되는 문제가 있었음

### v9가 **바꾸지 않은 것** (내러티브 경로)
- `evaluateRules()` 함수 (line 375)는 `Map<String, int> scores`를 입력으로 받음
- 이 `scores`는 **메트릭 z-score 버킷**이지 **attribute 점수가 아님**
- Rule 조건은 `s['gonialAngle']! >= 1` 처럼 **메트릭 버킷만 체크**
- v9가 바꾼 것은 **attribute 원점수 → 표시 점수**뿐
- 따라서 **v9는 rule 발동 개수에 이론적으로 영향을 미치지 않습니다**

### 그럼 왜 "엔진 업그레이드 후 짧아졌다"고 느끼는가?

v9 자체가 rule 개수를 줄이진 않았지만, **인지적 대비**를 키웠습니다:

- v8 시절: 점수 분포가 좁아서 모든 attribute가 6~7점대로 다닥다닥 붙어있음 → 유저가 "왜 텍스트는 2개뿐인가"에 둔감
- v9 이후: 점수 spread 보장으로 **최상위 attribute가 8, 9, 10점으로 선명하게** 보임 → "9점인 지능 얘긴 어디 갔나?"라는 불일치가 **시각적으로 도드라짐**
- **동일한 수의 rule이 발동하더라도, 유저 입장에서는 "막대는 강렬한데 글은 비어있다"라는 격차가 크게 느껴짐**

즉 v9는 증상의 **원인이 아니라 증상의 가시성 증폭기**입니다.
원래부터 존재하던 "평범한 얼굴은 rule 발동이 희박하다"는 구조적 한계가 v9의 강한 점수 대비 때문에 눈에 띄게 된 것입니다.

## 5. 해결 방향 (왜 fallback 텍스트인가)

### 고쳐야 하지 않는 것
- v9 calibration: 유저 요구사항 — 손대지 말 것
- Rule 조건 임계치: 느슨하게 풀면 v8 시절의 saturate 문제가 다시 나올 수 있음
- Rule 정의 자체: 이미 합리적으로 튜닝됨

### 고칠 수 있는 것: **report_assembler의 텍스트 소스를 두 갈래로**

```
attribute A에 대해
  ├── triggered rule 블록이 1개 이상 있으면  → 기존 그대로 사용
  └── triggered rule 블록이 0개이면         → NEW: score band fallback 텍스트
```

Fallback 텍스트 정의:
- `data/constants/attribute_fallback_blocks.dart` (신규) 추가
- attribute 10개 × score band 5단계 (매우 낮음 ≤ 6.0, 낮음 ≤ 7.0, 중간 ≤ 8.0, 높음 ≤ 9.0, 매우 높음 > 9.0)
- 짧은 2~3문장 설명 (기존 rule_text_blocks의 톤과 맞춤)
- **v9 점수를 단지 "읽기"만 하고 아무것도 수정하지 않음**

Assembler 변경 (`report_assembler.dart`):

```dart
// Before: triggered rule이 있는 attribute만 Map에 들어감
// After: Attribute.values 모두 순회, 각자 triggered 블록 or fallback 블록 선택

for (final attr in Attribute.values) {
  final triggered = matchedBlocks.where((b) => b.attribute == attr.name).toList();
  if (triggered.isNotEmpty) {
    sections[attr] = triggered.map((b) => b.bodyKo).join('\n');
  } else {
    final score = report.attributeScores[attr] ?? 7.0;
    sections[attr] = fallbackTextFor(attr, score);
  }
}
// 점수순 정렬 후 top N 선택
```

### 이 방식의 성질
| 요구사항 | 만족 여부 |
|---|---|
| v9 calibration 미변경 | ✅ 읽기만 함 |
| Rule 엔진 미변경 | ✅ 기존 triggered 블록이 있으면 그대로 사용 |
| 비어있는 attribute 소제목이 나오지 않음 | ✅ 모든 attribute가 triggered or fallback 중 하나를 반드시 가짐 |
| 막대 차트와 내러티브 일관성 | ✅ 막대가 높은 attribute엔 "매우 높음" 톤 텍스트, 낮은 attribute엔 "낮음" 톤 텍스트 |
| 평범한 얼굴도 충분한 분석 분량 | ✅ rule이 하나도 안 걸려도 10개 attribute × fallback으로 항상 확보 |

### Top-N 유지 여부
- `.take(5)`를 유지하면 항상 5개 소제목이 보장됨 (짧지만 균일)
- `.take(10)`으로 확장하면 풍부하지만 길어짐
- 권장: **top 5 유지** (막대 차트 상위 5개 + 하위 5개 대비가 깨끗함)
  또는 **top 8** (균형점)

## 6. 요약 한 문단

**"신뢰성/바람기만 보이는 현상"의 뿌리는 v9가 아니라, 내러티브가 "trigger된 rule"만으로 조립되는 이분법적 구조**입니다.
Rule 조건이 AND 결합된 보수적 컷오프라 평범한 얼굴은 대부분의 attribute에서 rule이 아예 발동하지 않고, 그 attribute 소제목은 시작부터 존재하지 못합니다.
v9는 이 문제를 만들진 않았지만, 점수 spread를 수학적으로 보장하면서 **막대는 강한데 글은 비어있는 불일치**를 시각적으로 증폭시켰습니다.
해결은 엔진 어떤 로직도 건드리지 않고, `report_assembler`가 **triggered 블록이 없는 attribute에 대해 score band fallback 텍스트를 주입**하도록 하는 것입니다. 이렇게 하면 v9 calibration은 그대로, rule 엔진은 그대로, 그러나 모든 attribute가 막대 점수와 어울리는 내러티브를 갖게 됩니다.

## 7. 쉬운 설명 (중학생 버전)

### 비유: 성적표 만드는 두 선생님

한 학생의 얼굴을 분석한다고 생각해 봅시다. 우리 앱 안에는 두 명의 선생님이 있어요.

**점수 선생님 (막대 차트 담당)**
- 10개 과목(지능, 신뢰성, 바람기, 인간관계, 재물운 등)마다 **무조건 점수를 매깁니다.**
- "이 학생은 얼굴 특징을 보니 지능 8.5점, 신뢰성 9.0점, 재물운 6.2점…" 이런 식으로요.
- 그래서 앱 화면의 **막대 그래프에는 언제나 10개 막대가 다 보입니다.**

**글쓰기 선생님 (분석 문장 담당)**
- 이 선생님은 "미리 정해진 레시피"만 사용해서 글을 씁니다.
- 레시피는 이런 식이에요: **"눈썹이 아주 굵고(기준 통과) + 턱선이 아주 강하면(기준 통과) → '리더십' 이야기 한 문단 쓴다"**
- **두 조건이 모두 맞아야** 레시피가 작동합니다. 하나만 맞으면 레시피는 발동하지 않아요.
- 그런데 학생 얼굴이 "그럭저럭 평범한" 쪽이면, 어느 기준도 넘지 못해서 **레시피가 하나도 발동 안 할 수 있습니다.**
- 발동한 레시피가 없는 과목은 **그 과목의 소제목 자체가 글에 나오지 않아요.**

### 그래서 생긴 문제

현재 앱에서는:
- **막대 선생님**은 10개 과목 점수를 다 보여줍니다 (예: 지능 9.0, 신뢰성 8.5, 재물운 6.2…)
- **글쓰기 선생님**은 그 중 레시피가 발동한 과목만 글로 씁니다 (예: 신뢰성, 바람기 두 개만 발동)
- 유저가 보기엔: "어? 지능 9점이라고 막대가 높은데 지능 얘기는 한 줄도 없네?"

### v9 업데이트가 바꾼 것

우리 앱은 최근 "v9 calibration"이라는 업데이트를 했어요. 이게 한 일은 딱 하나입니다:
- **막대 선생님의 점수 매기는 방식을 더 똑똑하게 만든 것.**
- 예전엔 10개 과목이 다 6~7점대에 다닥다닥 붙어 있어서 누구를 봐도 비슷해 보였어요.
- 이제는 가장 잘하는 과목은 8점 이상, 가장 못하는 과목은 7점 이하로 **점수 차이를 뚜렷하게** 만들어 줘요.

**중요**: v9는 **글쓰기 선생님은 전혀 건드리지 않았어요.** 레시피도 그대로, 발동 조건도 그대로입니다.

### 그런데 왜 유저는 "분석이 짧아졌다"고 느끼는가?

- v9 전: 모든 과목 점수가 고만고만해서 유저가 글과 막대의 불일치를 잘 못 느꼈어요.
- v9 후: 막대가 뚜렷해지니까 "저기 9점짜리 과목 있는데 글이 왜 없지?"라는 **의문이 눈에 띄게 커진 것**이에요.
- **실제로 글이 짧아진 게 아니라, 짧은 게 티가 나게 된 것.**

### 어떻게 고칠 것인가

v9 점수 매기는 방식은 건드리지 않고, **글쓰기 선생님에게 "예비 레시피"를 하나 더 줍니다:**

> "레시피가 하나도 발동 안 한 과목이 있으면, 막대 선생님이 매긴 점수를 보고 자동으로 기본 설명을 써라."
>
> - 9점 이상 → "이 부분이 매우 뛰어나 보입니다. ..." 같은 높은 톤 설명
> - 7~8점 → "평균 이상으로 보입니다. ..."
> - 5~6점 → "이 영역은 평범한 편입니다. ..."
> - 그 외 낮은 점수도 비슷하게

이렇게 하면:
- 원래 발동되던 레시피는 **그대로** 사용됩니다.
- 발동 안 된 과목은 **점수에 어울리는 예비 설명**이 대신 들어갑니다.
- 막대에 9점이 있는 과목은 "매우 뛰어나다" 톤의 글이, 6점인 과목은 "평범하다" 톤의 글이 나옵니다.
- **막대와 글이 서로 맞게** 됩니다.

### 한 줄 정리

**"막대는 10개 다 보이는데 글은 2개만 보이는 이상함"**은 **"글쓰기 선생님의 레시피가 너무 엄격해서 평범한 얼굴엔 레시피가 거의 발동하지 않기 때문"**입니다. v9는 원인이 아니라 이 문제를 더 잘 보이게 만들었을 뿐이고, 해결책은 **"레시피가 없는 과목엔 점수를 보고 기본 설명을 대신 써주자"**입니다.

## 8. 왜 점수 트랙과 내러티브 트랙을 굳이 2원화했는가

"둘로 나뉘어 있는 것 자체가 문제 아닌가? 하나로 합치면 안 되나?"라는 의문이 자연스럽게 들 수 있습니다. 답: **2원화는 버그가 아니라 의도적 디자인**입니다. 두 트랙은 각자 **다른 종류의 질문에 답하기 위해** 태어났고, 서로 겹치는 것처럼 보여도 실제로는 답하는 영역이 다릅니다.

### 8.1 점수 트랙이 답하는 질문
- "이 얼굴의 10개 특성을 **한눈에 비교**하고 싶다"
- "**다른 사람과 비교**하려면 숫자가 필요하다" — 궁합 엔진(`compatibility_engine.dart`)이 얼굴 A와 B의 attribute 벡터를 비교해서 궁합 점수를 계산합니다. 이때 "rule이 발동했느냐"로는 수치 비교가 불가능합니다.
- "UI에 **막대 그래프**를 그리려면 연속값이 필요하다"
- 요구 성질: **빠짐 없음, 비교 가능, 연속적(continuous)**

→ 그래서 **메트릭 가중합 → v9 정규화 → 5.0~10.0 연속 점수**라는 형태가 자연스럽게 선택됐습니다. 모든 얼굴의 모든 attribute가 항상 숫자를 가지게 되는 것이 이 트랙의 본질입니다.

### 8.2 내러티브 트랙이 답하는 질문
- "이 얼굴이 **어떻게 특별한가**를 한국어 관상학 문장으로 설명하고 싶다"
- 전통 관상학의 해석은 **단일 특징이 아니라 조합에서 나옵니다** — "턱이 강하면서 동시에 눈매가 날카로우면 … 한다"와 같이 두 메트릭 이상의 **AND 결합 패턴**이 의미를 만듭니다.
- Rule 조건이 `gonialAngle >= 1 && eyeCanthalTilt >= 1`처럼 AND로 묶인 이유가 바로 이것입니다.
- 점수로는 "리더십 8.3점"이라고 말할 수 있지만, 그 점수를 **왜** 받았는지, **어떤 조합 때문인지**는 점수만으로는 표현할 수 없습니다.
- 요구 성질: **구체적, 패턴 기반, 조건이 맞았을 때만 의미 있음(discrete)**

→ 그래서 **메트릭 버킷 조건 → triggered rule → text block → 조립**이라는 형태가 선택됐습니다. 이 트랙의 본질은 **특별한 조합이 있을 때만 무언가 말한다**는 것입니다.

### 8.3 왜 합치면 안 되는가 — 합치는 세 가지 방법과 그 대가

| 합치는 방법 | 잃는 것 |
|---|---|
| **① 점수만으로 텍스트 자동 생성 (rule 제거)** | 관상학의 조합 해석이 통째로 사라짐. "당신의 지능은 8.3점입니다" 식의 로봇 톤 생성문만 남아 전통 관상학의 맛이 증발. 또한 "어느 조합이 이 점수를 만들었는가"가 말해지지 못함 |
| **② Rule만으로 차트 대체 (점수 제거)** | 평범한 얼굴은 rule이 0~2개 발동 → 차트에 막대 1~2개만 존재. 10개 특성 비교 UI 불가능. 궁합 엔진이 점수 벡터를 요구하므로 **궁합 기능 자체가 붕괴** |
| **③ 점수를 rule 트리거로 전용 (score ≥ 8 → rule 발동)** | rule의 장점인 "두 메트릭 조합 해석"을 포기해야 함. 점수가 낮아도 의미 있는 조합(예: "약한 턱 + 얇은 입술" 같은)이 있을 수 있는데 그것들이 모조리 소실됨. 또한 v9의 rank-aware 정규화는 최상위 attribute가 항상 8 이상이므로 결국 **상위 1~2개만 rule이 붙게 됨** — 현재 증상이 재현 |

### 8.4 두 트랙의 관계 — 표로 정리

| | 점수 트랙 | 내러티브 트랙 |
|---|---|---|
| **입력** | 메트릭 raw 값 (double) | 메트릭 정수 버킷 (int) |
| **수학** | 가중합 → percentile → rank-blend → 5~10 | 정수 비교 AND 체인 |
| **출력 단위** | attribute 10개 모두 (빠짐 없음) | 발동된 rule만 (희박할 수 있음) |
| **답하는 질문** | "이 얼굴은 10개 특성 중 어디가 강하고 약한가?" | "이 얼굴의 어떤 조합이 특별한가?" |
| **UI 용도** | 막대 차트, 궁합 엔진 | 한국어 서술문 |
| **성질** | 정량, 연속, 비교 가능 | 정성, 이산, 패턴 기반 |
| **v9 영향** | 받음 (정규화 방식 변경) | 받지 않음 (rule 엔진은 독립) |

두 트랙은 **동일한 12개 메트릭에서 갈라져 나와** 서로 독립적으로 결과를 만들어냅니다. 합치지 않은 이유는 각자가 잘하는 일이 본질적으로 다르기 때문입니다: **"비교 가능한 숫자"**와 **"패턴 기반 해석"**은 서로 다른 정보 형태이고, 각자 잘하는 걸 시키는 게 현재 구조입니다.

### 8.5 Fallback 텍스트는 "합치기"가 아니라 "한쪽 방향 브리지"

제가 §5에서 제안한 fallback 텍스트 방식은 **2원화를 해체하는 것이 아닙니다**. 두 트랙을 그대로 분리한 채 유지하고, 다만 내러티브 트랙의 **빈 자리를 점수 트랙에서 읽어온 값으로 메꾸는 얕은 브리지**입니다.

```
[점수 트랙]          [내러티브 트랙]
     │                    │
     │                    ▼
     │              rule triggered?
     │              │          │
     │              YES        NO
     │              │          │
     │              ▼          ▼
     │         기존 블록    ◄─ (fallback 트리거)
     │                        │
     └────── 점수 읽기 ───────►│
                              ▼
                     score band 텍스트 주입
```

- **단방향**: 점수 트랙 → 내러티브 트랙. 반대 방향은 없음.
- **읽기 전용**: 점수 트랙의 값을 **수정하지 않고 단지 읽기만** 합니다. v9 calibration은 **완전히 그대로** 유지됩니다.
- **조건부**: rule 기반 블록이 있는 attribute에는 절대 개입하지 않습니다. rule이 0개일 때만 발동합니다.
- **트랙의 독립성 유지**: 궁합 엔진, 막대 차트 등 점수 트랙을 쓰는 다른 소비자들은 아무 영향도 받지 않습니다.

### 8.6 그래서 왜 2원화인가 — 한 문단 요약

점수 트랙은 **"10개 특성을 빠짐 없이 비교 가능한 숫자로 표현해야 한다"**는 요구 때문에 존재하고, 내러티브 트랙은 **"전통 관상학의 조합 해석을 한국어 문장으로 전달해야 한다"**는 요구 때문에 존재합니다. 두 요구는 서로 다른 수학적 성질(연속 vs 이산)과 서로 다른 완전성 보장(전부 vs 있으면 있고 없으면 없음)을 필요로 하므로, 하나의 엔진으로 양쪽을 모두 잘하는 것은 **원리적으로 어렵습니다**. 그래서 두 트랙을 분리하고, 각자 잘하는 일에 집중시킨 뒤, 필요할 때 얕은 브리지(fallback 텍스트)로 유저 경험의 빈틈만 메꾸는 것이 가장 비용 대비 효과가 좋은 디자인입니다.
